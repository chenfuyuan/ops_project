# 需求优先级排序

基于项目计划文档和代码实现现状，帮助用户快速判断当前最值得优先推进的需求。

**Input**: 可选参数。可传入关注范围、阶段名、需求关键词或空参数。无论是否传参，本命令都只输出需求优先级排序与推荐理由。

<HARD-GATE>
本命令只进行只读分析，不生成或修改任何文档，不编写代码，不启动 OpenSpec change，不调用其他工作流命令。
</HARD-GATE>

## 唯一目标

输出当前候选需求的优先级排序，并说明每个候选的推荐理由。

不要进行以下操作：

- 不进入需求拆解
- 不澄清边界
- 不拆子任务
- 不输出交接摘要
- 不提示 `/opsx:pre-design`、`/opsx:new` 或其他下一步命令
- 不询问用户是否接受推荐

## 分析步骤

### 1. 读取项目全景

- 读取 `novel_docs/project_plan/项目规划路线图总览.md`，理解阶段依赖关系。
- 读取相关阶段的 `阶段总览.md`，了解入口/出口条件。
- 读取相关阶段的 `需求索引.md`，收集需求状态、优先级和依赖。
- 如果用户指定了阶段或关键词，只聚焦匹配范围；如果未指定，则覆盖全部阶段。

### 2. 检查实现状态

- 优先读取 `openspec/实现状态.md`，获取已知实现状态。
- 仅在实现状态缺失或明显不完整时，才检查 `app/` 和 `openspec/changes/archive/`。
- 如果发现文档状态与代码现状不一致，只将其作为排序理由的一部分简要说明。

### 3. 排序候选需求

按以下优先级综合排序：

1. **阶段顺序** — 靠前阶段优先。
2. **前置满足度** — 依赖已完成或接近完成的需求优先。
3. **需求优先级** — P0 > P1 > P2。
4. **解锁效果** — 完成后能解锁更多后续需求的优先。
5. **实现状态** — 已具备基础设施或已有部分实现的需求优先于完全未开始的大型需求。

默认输出 2-3 个最优候选；如果候选很少，可以少于 2 个。

## 输出格式

只输出以下结构，不添加其他章节、问题、交接摘要或下一步建议：

```markdown
## 需求优先级

1. <需求名称>（<阶段>，<P0/P1/P2>）
   - 推荐理由：<为什么排第一，包括前置条件、解锁效果、实现状态等>

2. <需求名称>（<阶段>，<P0/P1/P2>）
   - 推荐理由：<为什么排在这里>

3. <需求名称>（<阶段>，<P0/P1/P2>）
   - 推荐理由：<为什么排在这里>
```

如果没有可推进需求，输出：

```markdown
## 需求优先级

暂无明确可推进需求。

- 推荐理由：<说明阻塞原因或缺失信息>
```

## 文档探索指引

```text
实现状态：openspec/实现状态.md
路线图入口：novel_docs/project_plan/项目规划路线图总览.md
阶段目录：novel_docs/project_plan/phases/{mvp,consistency,narrative,style,automation,saas}/
阶段总览：<phase>/阶段总览.md
需求索引：<phase>/需求索引.md
代码现状：app/ 目录
已完成变更：openspec/changes/archive/
进行中变更：openspec/changes/（非 archive 子目录）
```

## 严格限制

- 只输出需求优先级和推荐理由。
- 不输出复杂度评估，除非它直接影响排序理由。
- 不输出需求拆解、实现路径、子任务列表或交接摘要。
- 不提示下一步命令。
- 不修改任何文件。
- 不启动任何 OpenSpec 工作流。

# 测试规则

本文件定义本仓库中 AI 生成和补充测试时应遵循的默认测试策略。

## 总体原则
- 优先让测试边界与架构边界一致。
- 业务核心优先通过抽象 port 做单元测试。
- 边界适配、协议映射和基础设施集成优先通过集成测试验证。
- 关键结构约束应尽量通过架构测试或等价检查自动化守护。

## 单元测试
### 重点对象
优先为以下文件补单元测试：
- `service.py`
- `rules.py`
- `entities.py`

### 测试方式
- 通过 `ports` 边界注入 fake / stub / mock。
- 不依赖真实三方服务。
- 不依赖真实网络。
- 关注业务规则、业务编排和领域对象行为是否正确。

### AI 编写单测时应遵循
- 优先测试业务输入、业务输出和业务决策。
- 不要在单元测试里验证第三方 SDK 的行为。
- 不要把集成问题伪装成单元测试。
- 如果一个 `service.py` 需要真实数据库、真实 HTTP 或真实队列才能测，通常说明边界设计不对，或测试层级选错了。

## 集成测试
### 重点对象
优先为以下区域补集成测试：
- `infrastructure/`
- `capabilities/infrastructure/`
- `interfaces/`

### 验证内容
集成测试重点验证：
- 协议映射是否正确
- 外部依赖连接和交互是否稳定
- 数据映射与异常转换是否符合预期
- timeout、retry、auth、错误翻译等横切策略是否符合预期

### AI 编写集成测试时应遵循
- 重点覆盖 adapter、repository、协议边界和外部依赖映射。
- 测试目标应是“边界是否正确工作”，而不是重复单元测试中的业务分支。
- 当某段逻辑的核心风险来自协议、存储、序列化或 provider 差异时，应优先考虑集成测试。

## 端到端测试
### 目标
端到端测试应从 `interfaces` 入口验证完整调用链。

### 适用场景
- 需要确认从外部入口到业务编排再到边界适配的整体路径是否正确。
- 需要验证关键 golden path 是否在真实装配下可运行。

### AI 编写端到端测试时应遵循
- 聚焦少量关键链路，不要把所有细节都堆到 E2E。
- 让 E2E 测试承担“系统是否打通”的职责，而不是替代所有下层测试。

## 架构测试
### 目标
架构测试用于自动校验结构约束，防止代码逐步破坏边界。

### 应优先检查的规则
- 禁止 `service.py` 直接 import 供应商 SDK。
- 禁止 `service.py` 直接依赖 ORM、HTTP client 或其他具体实现。
- 禁止 `interfaces` 直接依赖业务侧 `infrastructure`。
- 禁止 `capabilities` 依赖 `business`。
- 禁止 `shared` 引入业务语义模块。
- 检查业务调用是否遵循 `service -> ports -> infrastructure` 链路。

### AI 编写架构测试时应遵循
- 优先把高价值、稳定的边界规则自动化。
- 不要把过于脆弱、容易因重命名而频繁误报的规则写得过细。
- 架构测试应服务于边界守护，而不是制造维护噪音。

## 分层选择规则
当需要判断测试写在哪一层时，可按下面方式判断：
- 测业务规则和业务编排：单元测试
- 测 adapter、协议映射、存储交互、外部依赖集成：集成测试
- 测完整入口到出口链路：端到端测试
- 测 import 方向和结构边界：架构测试

## 生成测试时的优先顺序
当 AI 需要为新代码补测试时，默认优先顺序是：
1. 先补业务核心的单元测试
2. 再补边界适配的集成测试
3. 对关键路径补少量端到端测试
4. 对稳定边界补架构测试

## 常见误区
- 用大量 E2E 测试替代本该写的单元测试和集成测试。
- 在单元测试里偷偷依赖真实数据库、真实网络或真实外部服务。
- 只测 happy path，不测映射失败、异常翻译或关键边界错误。
- 明明是结构约束问题，却只靠人工 code review，不做自动化检查。

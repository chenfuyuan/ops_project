## MODIFIED Requirements

### Requirement: 应用代码最小 CI 实践必须定义默认覆盖对象
系统 MUST 为应用代码定义一套最小 CI 实践，并将默认覆盖对象限制为 `app/` 及其直接相关测试。该实践 MUST 以应用代码准入为目标，而不是扩展到文档门禁、受控部署或具体平台配置。后续文档与实现 MUST 能明确区分哪些检查属于应用代码最小 CI 范围，哪些内容超出本次 change 边界。该 capability 后续可以被映射到 GitHub Actions 等具体 CI 平台，但这种映射 MUST 只作为执行承载层存在，不得改变默认覆盖范围与职责边界。

#### Scenario: 以应用代码为默认覆盖范围
- **WHEN** 团队查阅该 change 的提案、设计或任务文档
- **THEN** 可以明确看到最小 CI 默认覆盖 `app/` 与其直接相关的测试目录
- **AND** 不会将 docs/spec 一致性治理、CD 或平台配置实现误认为本次默认范围

#### Scenario: 非目标不会被纳入应用代码最小 CI
- **WHEN** 有人尝试在该 capability 中加入文档门禁、受控部署或平台专属流水线配置
- **THEN** 规格能够明确判定这些内容超出应用代码最小 CI 实践边界

#### Scenario: 平台映射不会改写应用代码最小 CI 边界
- **WHEN** 团队将该 capability 映射到 GitHub Actions 等具体平台
- **THEN** 平台配置仍只承接应用代码准入所需的最小检查范围
- **AND** 不会因为平台落地而把文档治理、CD 或其他非目标内容并入该 capability

### Requirement: 应用代码最小 CI 必须采用三层最小流水线模型
系统 MUST 将应用代码最小 CI 组织为三层最小流水线模型：静态与结构校验层、测试校验层、可运行性与构建确认层。该模型 MUST 用于表达默认校验顺序、职责分工与阻断关系，并且 MUST 保持平台中立，不依赖任何具体 CI 平台语法或平台专属功能。后续若将该模型映射到 GitHub Actions 等具体平台，MUST 保持三层职责与前层失败阻断后层的顺序语义不变。

#### Scenario: 三层模型用于表达最小 CI 结构
- **WHEN** 团队根据该 capability 编写后续设计或实现任务
- **THEN** 可以看到静态与结构校验、测试校验、可运行性与构建确认三层职责被分别表达
- **AND** 不会把所有检查混成一组无分层的命令列表

#### Scenario: 三层模型可映射到平台但不绑定平台
- **WHEN** 团队后续需要把该最小 CI 实践映射到具体流水线平台
- **THEN** 可以在不改变三层职责边界的前提下完成阶段或作业映射
- **AND** 规格本身不会写入 GitHub Actions、GitLab CI、Jenkins 等平台专属语法

#### Scenario: 平台映射保持前层阻断后层的顺序语义
- **WHEN** 团队在 GitHub Actions 等平台实现三层最小流水线模型
- **THEN** 前一层的阻断失败会阻止后一层继续执行
- **AND** 平台实现不会破坏该 capability 定义的默认顺序与职责边界

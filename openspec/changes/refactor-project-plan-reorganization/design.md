## Context

当前 `novel_docs/project_plan/` 中同时存在总览、架构、数据模型、阶段说明和专题设计等多类文档，主入口已经承载了超出“前期规划”阶段的职责。根据 `pre_design.md`，本次变更的核心不是继续补齐方案，而是重建规划文档的职责边界：将 `project_plan` 收敛为路线图导航与需求拆分入口，把设计推迟到未来真正启动单个需求时，再通过 OpenSpec change 展开。

本次设计还受到几个硬约束：phase 只能作为路线图分组，不能成为执行单元；单张需求卡片必须成为未来进入 change 的最小稳定单元；旧规划资产必须整体归档，不能与新主入口长期混放；在当前阶段不得提前引入接口设计、数据模型、模块拆分或任务拆解。

## Goals / Non-Goals

**Goals:**
- 让 `novel_docs/project_plan/` 成为唯一的前期规划入口。
- 将目录重组为 `00-overview.md + phases/<phase>/...` 的 phase-first 结构。
- 为每个 phase 建立轻量标准文档集，仅包含 phase 概览、需求索引和单需求卡片。
- 明确旧规划文档的统一归档位置，避免新旧双入口并存。
- 让单张需求卡片足以表达“为什么做、做什么、不做什么”，并可作为未来创建单个 change 的上游输入。

**Non-Goals:**
- 不在本次变更中提前创建任何面向单需求的 `openspec/changes/*`。
- 不输出接口定义、数据模型、模块结构、详细流程或实现步骤。
- 不把 phase 文档扩展回“大设计总纲”或执行方案容器。
- 不改变运行时代码、API、数据库或工程依赖。

## Decisions

### 决策 1：以 `phase-first` 重建 `project_plan/` 主结构
- 选择：使用 `novel_docs/project_plan/00-overview.md` 作为总导航入口，并在 `phases/` 下按 phase 组织文档。
- 原因：这能保留路线图视角，同时避免旧结构中“按主题堆叠设计文档”导致的阅读路径混乱。
- 备选方案：继续在顶层平铺 architecture、data-model、core-services、phase 文档。
- 放弃原因：会继续强化“先做完整设计”的错误心智模型，且无法清晰表达当前与历史资产的边界。

### 决策 2：phase 只保留轻量标准文档集
- 选择：每个 phase 只保留 `00-overview.md`、`requirements.md` 与 `requirements/<slug>.md`。
- 原因：通过固定结构限制 phase 文档的职责，确保 phase 是路线图容器而不是实现容器。
- 备选方案：允许 phase 下继续保留专题设计、模块设计或步骤拆解文档。
- 放弃原因：会再次把设计和执行细节提前注入规划阶段，破坏 `pre_design` 设定的边界。

### 决策 3：将单需求卡片定义为最小稳定单元
- 选择：单张卡片固定表达 Goal、User Value、Success Criteria、Scope、Non-goals、Dependencies / Prerequisites、Notes。
- 原因：该信息密度足以支撑未来进入单个 OpenSpec change，又不会滑向准设计文档。
- 备选方案：只保留一句话需求索引，或直接把卡片写成设计草案。
- 放弃原因：前者信息不足，后者超出本次变更目标，都会削弱卡片作为稳定上游输入的作用。

### 决策 4：旧规划资产统一归档并退出主路径
- 选择：将当前 `novel_docs/project_plan/` 下的旧文档整体移动到独立 archive 位置，新的主路径只承载当前路线图与需求卡片。
- 原因：必须消除双入口，防止团队继续基于旧设计文档开展后续规划。
- 备选方案：新旧结构长期并存，或对旧文档做局部修补。
- 放弃原因：都会保留歧义入口，降低新结构的约束力，并延续原有问题。

### 决策 5：在文档层明确“卡片 -> future change”的工作流映射
- 选择：在 overview、phase 文档和卡片职责中明确，未来真正启动需求时，应从单张需求卡片进入 OpenSpec change。
- 原因：这能把本次文档重整与未来实施衔接起来，同时避免现在就提前进入设计和任务拆解。
- 备选方案：现在就为每张卡片预生成 proposal/design/tasks。
- 放弃原因：与本次“先整理需求，不做设计”的目标冲突。

## Risks / Trade-offs

- [风险] 旧文档信息量较大，归档后可能有人担心内容丢失。 → Mitigation：统一归档而不是删除，并在新入口中明确旧资产只作为历史参考。
- [风险] 需求卡片过轻，可能被误解为信息不足。 → Mitigation：在固定字段中要求清晰写出目标、价值、范围和非目标，保证进入 change 前的信息完整度。
- [风险] phase 文档容易再次被扩写成设计文档。 → Mitigation：在 overview、requirements 和卡片规范中显式写明禁止事项，并通过 spec 约束其内容边界。
- [权衡] 本次不提前展开设计，会延后部分技术细节讨论。 → Mitigation：接受这种延后，把技术决策留到真正选定需求并创建 change 时处理，以换取更清晰的规划边界。

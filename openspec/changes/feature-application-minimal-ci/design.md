## Context

当前仓库已经通过 `docs/cicd_baseline.md` 与 `openspec/specs/cicd-baseline/spec.md` 建立了平台中立的 CI/CD 主基线，但这层定义主要关注阶段边界、默认门禁组成、可发布制品与受控部署范围，尚未进一步下沉为“应用代码默认应经过哪些最小 CI 实践”的落地约束。

仓库当前已经具备应用代码与测试骨架，包括 `app/` 下的 API、Worker、bootstrap、interfaces 代码，以及 `tests/unit/`、`tests/integration/`、`tests/architecture/` 等测试分层基础。这意味着项目已经具备形成应用代码最小 CI 实践的客观条件，但仍缺少一份能够把这些现状与主基线衔接起来的设计文档。

本设计的约束来自 `pre_design.md`：
- 只覆盖应用代码 CI，不扩展到 CD。
- 只定义最小流水线实践，不绑定具体平台配置。
- 默认覆盖对象以 `app/` 与其直接相关测试为中心。
- 架构/边界检查必须作为默认阻断项。
- 不得把企业平台能力、制品治理能力或未落地资产写成仓库现状。

## Goals / Non-Goals

**Goals:**
- 为应用代码定义一套平台中立、可映射到后续流水线实现的最小 CI 实践结构。
- 明确最小 CI 的三层模型：静态与结构校验层、测试校验层、可运行性/构建确认层。
- 明确默认覆盖对象、阻断门禁与层间关系，使后续 `tasks.md` 能据此拆解实现工作。
- 与现有 `cicd-baseline`、架构规则、测试规则和 `uv` 基线保持一致。
- 明确主基线与下游应用代码实践之间的职责分层，避免范围漂移。

**Non-Goals:**
- 不设计 GitHub Actions、GitLab CI、Jenkins 等平台配置。
- 不定义文档门禁、CD、发布后验证、回滚流程或多环境晋升策略。
- 不重构现有业务架构、测试体系或仓库目录结构。
- 不把镜像扫描、制品签名、审批链、企业平台集成等能力纳入本次最小实践。

## Decisions

### 决策 1：采用“三层最小 CI”结构，而不是单层命令清单
**选择：** 将应用代码最小 CI 设计为三层：静态与结构校验层、测试校验层、可运行性/构建确认层。

**原因：**
- 这比单纯罗列命令更接近后续平台无关的 stage/job 设计。
- 能清晰表达不同失败类型的语义，例如结构性违规、测试失败、装配不可运行。
- 与 `pre_design.md` 中已确认的推荐方案保持一致。

**备选方案：**
- 两层模型（检查层 + 测试层）：表达更短，但会弱化构建确认职责。
- 端到端流水线骨架优先：更贴近平台实现，但过早引入平台思维。

### 决策 2：覆盖对象限制为 `app/` 与其直接相关测试
**选择：** 最小 CI 默认覆盖 `app/` 及其直接相关的 `tests/unit/`、`tests/integration/`、`tests/architecture/` 范围。

**原因：**
- 这是本次 change 已确认的核心边界。
- 能把应用代码准入条件与文档治理、发布治理、平台配置治理解耦。
- 与现有仓库实际结构一致，便于后续任务直接落到代码和测试目录。

**备选方案：**
- 同时纳入 docs/spec 一致性：更接近完整默认 CI，但超出本次范围。
- 扩展到全仓库：会模糊“应用代码最小实践”的主题。

### 决策 3：架构/边界检查属于默认阻断门禁
**选择：** 将架构/边界检查作为静态与结构校验层中的默认阻断项。

**原因：**
- 现有 `cicd-baseline` 已明确该类检查不得降级为提示。
- 仓库的高价值约束正是边界与依赖方向，例如 `service.py -> ports.py -> infrastructure/*`。
- 若不在最小 CI 中阻断，最容易出现“代码能跑但持续侵蚀架构边界”的偏差。

**备选方案：**
- 仅作为提示信息：不符合既有基线。
- 放到独立增强阶段：会破坏“最小准入条件”的定位。

### 决策 4：测试校验层采用分层测试最小集合，而不是重型全链路优先
**选择：** 测试校验层以单元测试、已纳入默认准入的集成测试、架构测试为主，不要求用大规模 E2E 充当默认门禁。

**原因：**
- 与 `ai_docs/rules/testing_rules.md` 的测试分层一致。
- 能保证默认 CI 的可执行性与稳定性，避免让最小实践被过重测试拖垮。
- 更适合当前仓库已有测试基础。

**备选方案：**
- E2E 优先：信号更完整，但成本更高且不符合当前最小实践目标。
- 只跑单测：对边界适配与结构守护覆盖不足。

### 决策 5：可运行性/构建确认层只验证最小可运行前提，不承担制品发布职责
**选择：** 第三层只负责必要构建校验、导入/装配可运行性确认，不生成完整发布制品，也不承接 CD。

**原因：**
- 与 `pre_design.md` 中“只做 CI，不做 CD”的边界一致。
- 保留 `cicd-baseline` 中“可发布制品”作为上层治理概念，不在本次 change 中误写为具体实现要求。
- 让后续平台实现可以在不破坏边界的前提下继续扩展。

**备选方案：**
- 直接要求产出发布制品：会把范围推进到交付闭环甚至 CD。
- 完全不做运行性确认：又会让最小 CI 缺少“是否具备进入后续交付阶段的基本条件”信号。

## Risks / Trade-offs

- **[范围回弹到完整 CI/CD]** → 通过 specs 与 tasks 明确限定为“应用代码最小 CI 实践”，任何文档门禁、CD、平台配置都作为非目标排除。
- **[设计过于抽象，无法指导实现]** → 在三层模型中明确每层目标、覆盖对象、阻断语义与后续任务拆分方向。
- **[设计过于接近平台实现，破坏平台中立]** → 允许使用 stage/job 语言描述层间映射，但禁止出现平台专属语法与平台功能假设。
- **[默认门禁过重，影响最小实践稳定性]** → 将测试层控制在单元、已纳入默认准入的集成测试和架构测试的最小集合内，不用重型 E2E 兜底。
- **[主基线与下游实践职责重叠]** → 在 `cicd-baseline` delta spec 中显式补充两者的分层关系，避免后续文档重复定义。

## Migration Plan

1. 先通过 proposal 与 specs 固化“应用代码最小 CI 实践”的需求边界，以及它与 `cicd-baseline` 的职责分层。
2. 基于本 design 生成 `tasks.md`，把三层最小 CI 所需的文档、测试、校验项与命令基线拆成可执行任务。
3. 在实现阶段，仅落实最小 CI 所需资产与校验逻辑，不同时引入平台专属流水线配置。
4. 如后续需要映射到具体 CI 平台，应在不改变本设计边界的前提下，另行补充平台实现层任务或 change。
5. 若实践过程中发现某项检查超出最小实践稳定范围，应优先收窄到可稳定执行的最小集合，而不是扩展到更大范围。

**回滚策略：**
- 本次 design 本身不涉及运行时部署变更。
- 如果后续实现证明某项最小门禁不稳定，应回退该项落地资产，同时保留本设计对边界的约束，并在独立变更中重新评估扩展路径。

## Open Questions

- “必要的依赖与安全最低检查”在当前仓库中应收敛到哪些具体可执行检查，仍需要在 tasks 与实现阶段结合现有工具链进一步确认。
- 可运行性/构建确认层采用哪一种最小验证方式最合适（例如导入校验、启动校验、装配校验的组合），需要在不突破最小实践范围的前提下进一步细化。
- 哪些现有集成测试应被视为默认准入集合的一部分，仍需要结合当前测试稳定性与执行成本做最终收敛。

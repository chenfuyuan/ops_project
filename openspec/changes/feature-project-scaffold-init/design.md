## 背景（Context）

当前仓库尚未形成应用代码，只存在参考脚手架文档与 OpenSpec 变更文档。本次变更要把仓库初始化成一个“最小可运行、边界清晰优先”的模块化单体骨架，为后续业务模块接入提供统一起点。

设计受到以下硬约束：

- 顶层结构必须围绕 `app/modules/<module_name>/...` 组织未来业务，而不是先建立全局横向分层。
- 模块之间只能通过 `api/contracts.py` 与 `api/facade.py` 通信。
- `app/shared/` 只能承载稳定、无业务语义的公共能力。
- `app/bootstrap/` 是集中装配层，负责应用启动、依赖装配、模块注册扩展点。
- 当前阶段只要求应用可启动并暴露健康检查，不接数据库，不引入真实业务模块。

## 目标 / 非目标（Goals / Non-Goals）

**Goals：**
- 初始化最小可运行的 FastAPI 项目骨架。
- 固化 `app/main.py`、`bootstrap`、`interfaces/http`、`shared`、`modules`、`tests` 的职责边界。
- 提供清晰的模块模板与模块接入约定，作为后续模块扩展入口。
- 建立最基础的工程配置，使项目能作为后续实现的稳定起点。

**Non-Goals：**
- 不接入数据库、Alembic、缓存、消息队列或其他扩展基础设施。
- 不创建任何真实业务模块。
- 不提供完整平台化能力，如 Docker Compose、完整 CI 流程或生产部署方案。

## 关键决策（Decisions）

### 1. 采用“边界优先的最小可运行骨架”而不是“启动优先 demo”

选择先落下最终希望长期保留的目录边界，再在该边界内补最小运行链路。这样可以避免后续在已有功能基础上回收结构，降低项目退化为普通分层单体的风险。

备选方案是先起一个最短路径 FastAPI demo，再逐步整理结构。该方案短期更快，但会把早期路径依赖带入后续开发，因此不采用。

### 2. 以 `app/main.py -> bootstrap -> interfaces/http` 作为最小启动链路

应用入口只负责创建或获取应用对象；启动装配逻辑集中在 `bootstrap/`；HTTP 路由和中间件只放在 `interfaces/http/`。这样能把运行入口、系统装配、接口适配三类职责分开，为后续引入 CLI、consumer 或模块注册留下清晰扩展位。

备选方案是把路由注册和应用创建都放在 `main.py`。这会让初始代码更短，但会把接口层和装配逻辑耦合在一起，因此不采用。

### 3. `app/modules/` 使用模板文件而不是示例业务模块

本次在 `app/modules/` 中只保留 README 与生成模板，模板定义标准结构：`api/`、`application/`、`domain/`、`infrastructure/`、`module.py`。这样可以传递接入规范，而不通过虚构模块误导未来领域划分。

备选方案是创建一个“示例空模块”。这看似更直观，但容易让示例实现被误认为标准实现，甚至被复制出不合适的边界，因此不采用。

### 4. `shared` 只保留基础内核与基础设施占位

`shared/kernel` 放通用异常、基础类型、结果模型等稳定能力；`shared/infra` 放配置、日志、时钟等通用基础设施占位；`shared/events` 仅保留事件基类与事件总线基础接口占位。当前不允许把任何带业务语义的逻辑放入 `shared`。

备选方案是先把常见工具函数统一放进 `shared`，后续再收敛。该方案最容易导致共享层失控，因此不采用。

### 5. 最小交付范围只覆盖可启动与健康检查

本次把“可运行”的验收收敛为：应用能启动、能注册最小 HTTP 路由、能返回健康检查响应。数据库与其他基础设施只留结构扩展位，不出现在当前运行链路中。

备选方案是顺手补齐数据库连接、Alembic、Docker 等基础设施。虽然能提升完整度，但会直接突破 pre_design 约束并扩大 change 范围，因此不采用。

## 风险 / 权衡（Risks / Trade-offs）

- [骨架过轻，后续仍需补基础设施] → 通过在 design 与 tasks 中明确扩展位，确保后续补齐时不破坏当前边界。
- [占位文件较多，初看像“空项目”] → 用清晰的 README、模板与任务拆解说明每个目录的职责和后续接入方式。
- [没有示例业务模块，团队成员理解模板成本更高] → 在 `modules/README.md` 中把标准结构、依赖边界和注册方式写清楚，降低上手成本。
- [健康检查之外没有业务流验证] → 将最小运行目标限定为框架初始化验证，不把其误解为业务能力完备。

## 迁移计划（Migration Plan）

1. 创建应用目录、接口层、共享层、装配层与测试目录。
2. 建立 FastAPI 应用入口与 bootstrap 装配逻辑。
3. 注册健康检查路由与最小中间件占位，验证可启动。
4. 落下模块模板与说明文档，固定后续模块接入方式。
5. 补充基础工程配置文件，使仓库达到可继续实现状态。

回滚策略：当前仓库尚无既有应用代码，若变更方向错误，可整体回退此次初始化文件集，不涉及数据迁移或线上兼容问题。

## 待确认问题（Open Questions）

- `pyproject.toml` 中是否同时纳入类型检查或 pre-commit，当前 pre_design 未要求，因此默认不纳入本次范围。
- 是否在首版中提供 `interfaces/cli` 与 `interfaces/consumer` 的占位文件，还是只保留目录结构；该问题可在实施时依据“最小可运行”原则取最轻量方案。

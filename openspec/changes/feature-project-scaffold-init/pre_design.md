# Pre-design: 项目初始化为最小可运行的模块化单体骨架

## Problem framing

当前仓库基本为空，仅有 OpenSpec 配置与参考文档，需要完成首轮项目初始化。

本次初始化的目标不是交付任何业务模块，而是把仓库落成一个可长期演进的模块化单体骨架。其核心价值在于：从第一天开始固定模块边界、共享边界、系统装配边界，以及后续模块接入方式，避免项目在最初阶段退化为普通分层单体。

参考文件 `docs/新项目脚手架_纯骨架版_模块化单体.md` 已给出总体方向：

- 按模块纵向组织，而不是全局横向分层
- 模块之间只能通过公开 API 通信
- 共享层必须极小且无业务语义
- 单体运行，但边界按未来可拆分服务的标准设计

因此，这个 change 要解决的真实问题不是“如何尽快起一个 FastAPI 项目”，而是“如何在最小可运行前提下，建立一个不会破坏未来演进方向的初始工程骨架”。

## Goals

- 建立最小可运行的项目骨架。
- 固化模块边界、目录职责、模块接入约定与装配位置。
- 提供可启动的 FastAPI 应用与健康检查入口。
- 明确基础栈为 FastAPI、Pydantic、pytest、ruff。
- 在 `app/modules/` 下提供可复制的模块模板与约定文件，但不引入真实业务模块。

## Non-goals

- 不引入真实业务模块。
- 本次不接数据库，不初始化 SQLAlchemy/Alembic 运行链路。
- 本次不引入 Redis、消息队列、异步任务框架、工作流引擎等扩展基础设施。
- 本次不交付完整开发平台能力，例如 Docker Compose、本地依赖编排、完整 CI 配置等。
- 本次不自动推进 `proposal.md`、`design.md`、`tasks.md` 生成；当前产物只包含 `pre_design.md`。

## Requirement understanding

用户希望基于参考脚手架文档初始化项目，并明确选择以下方向：

- 初始化深度为“最小可运行”
- 数据库基础本次不接入
- 成功标准以“边界清晰优先”为主
- `app/modules/` 中不放真实业务模块，而是放生成模板文件
- 将 FastAPI、Pydantic、pytest、ruff 作为确定基础栈

这意味着本次输出必须同时满足两类要求：

1. **工程上可以启动**：至少应具备应用入口、配置入口、HTTP 路由注册与健康检查。
2. **结构上可持续扩展**：未来新增业务时，必须自然落入 `app/modules/<module_name>/...` 的边界，不需要对初始骨架做大改。

因此，单纯生成一个可跑的 FastAPI demo 并不满足目标；同样，单纯创建目录而没有可运行入口也不满足目标。需要的是二者之间的平衡：以架构边界为中心的最小可运行实现。

## Constraints / Invariants

- 目录组织必须以 `app/modules/<module_name>/...` 为未来模块主承载结构，而不是全局横向分层。
- 模块间调用规则必须在骨架阶段就固定：只允许依赖对方 `api/contracts.py` 与 `api/facade.py`。
- `app/shared/` 只能容纳稳定、无业务语义的横向基础能力，禁止演化为通用杂物层。
- `app/bootstrap/` 是系统装配层，负责模块注册、依赖组装、启动初始化；必须与业务模块边界分离。
- 当前应用需可运行，但“可运行”仅指能启动应用并暴露健康检查；不等价于已具备数据库、缓存、异步消费等完整基础设施。
- `app/modules/` 本次不落真实模块，仅落模板或约定，以避免示例代码反向污染边界设计。
- 生成模板的目的仅为统一后续模块接入方式，不得成为共享业务逻辑或公共实现的容器。
- 本次已确认不接数据库，因此任何涉及业务持久化、迁移脚本执行链路、ORM 初始化的设计都不得被后续文档默认补入当前 change。

## Architecture direction

推荐采用“边界优先的最小可运行模块化单体骨架”。

核心结构方向如下：

- `app/main.py`：应用启动入口
- `app/bootstrap/`：系统装配层，负责创建应用对象、初始化配置、注册路由与后续模块装配扩展点
- `app/interfaces/http/`：HTTP 输入输出层，当前至少包含健康检查路由、通用 schema、请求上下文中间件占位
- `app/shared/`：稳定公共能力，当前仅保留基础内核与基础设施占位
- `app/modules/`：未来业务模块承载目录，本次仅包含说明文档与生成模板
- `tests/`：预留 unit / integration / e2e 三层测试目录

当前仅实现最小 HTTP 路径：应用启动、基础配置、健康检查接口。数据库、缓存、消息消费等只保留目录边界或后续扩展位，不落运行实现。

`modules/README.md` 与生成模板共同承担“后续模块接入规范”的角色。相比创建示例空模块，模板方式更适合当前阶段，因为它定义结构但不暗示领域划分，也不会让样例代码被误当成标准实现。

## Options considered

### 方案 A：边界优先骨架（推荐）

先严格按参考文档建立 `modules / interfaces / shared / bootstrap / tests`，再补最小 FastAPI 启动链路与健康检查。

优点：

- 与参考文档目标一致
- 从初始状态就固定长期边界
- 后续新增真实模块时不需要重构顶层结构

代价：

- 首版会包含一些占位文件与说明文件
- 启动能力只覆盖最小路径，不追求平台完整度

### 方案 B：启动优先骨架

先做一个最短路径可跑的 FastAPI 项目，再逐步调整为模块化单体结构。

优点：

- 初始开发速度快
- 可以尽快获得运行反馈

代价：

- 很容易形成“先临时跑起来、后续再整理”的惯性
- 后续回收结构成本高，容易退化成全局分层单体

### 方案 C：工程规范优先骨架

先把 `pyproject.toml`、ruff、pytest、Makefile、脚本与目录标准化，再逐步接入应用结构。

优点：

- 工程规范感强
- 便于统一开发体验

代价：

- 不能优先回答“未来模块应该放哪、如何接入”的核心问题
- 对当前“边界清晰优先”的目标支撑较弱

### 推荐结论

采用方案 A。

本次 change 的首要价值是建立长期稳定的边界与约定，而不是最快速地得到一个可运行 demo。最小可运行能力只需要证明骨架是可启动的，不应反过来主导结构设计。

## Key decisions / Trade-offs

- 选择“边界清晰优先”而不是“启动体验优先”，接受首版脚手架中存在适度占位实现。
- 选择“最小可运行”而不是“开发全套基础”，因此数据库、Alembic、Docker 等仅保留扩展位，不在本次接入。
- 选择“生成模板文件”而不是“示例空模块”，以避免样例代码被误认为架构标准。
- 选择在 pre_design 中明确基础栈与目录职责，后续 `proposal/design/tasks` 只能在该边界内细化，不得推翻这些核心判断。

## Critical flows

### 启动流

1. 由 `app/main.py` 创建应用入口。
2. 调用 `app/bootstrap/` 中的装配逻辑。
3. bootstrap 初始化基础配置与应用对象。
4. bootstrap 注册 HTTP 路由与基础中间件。
5. 应用暴露健康检查接口，作为“最小可运行”验收路径。

### 后续模块接入流

1. 开发者基于 `app/modules/` 下的模板创建新模块目录。
2. 模块内部按 `api / application / domain / infrastructure / module.py` 分层。
3. 外部调用方只能通过该模块的 `api/contracts.py` 与 `api/facade.py` 访问能力。
4. 模块注册与装配在 `app/bootstrap/` 中集中处理，而不是散落到接口层或共享层。

## OpenSpec mapping

后续文档承接应遵循以下分工：

- `proposal.md`：说明为什么现在要初始化该骨架、交付范围、成功标准、非目标。
- `design.md`：展开目录结构、启动链路、模块模板、装配关系、接口层与共享层边界。
- `tasks.md`：拆解目录创建、基础文件初始化、FastAPI 入口、健康检查、模板文件、工程配置与最小测试。

后续文档不得自行扩展到以下内容：

- 数据库接线
- 真实业务模块
- 消息队列、Redis、异步任务系统
- 与当前最小可运行目标无关的平台化能力

## Generation guardrails

### 必须遵守

- 模块化单体边界与目录组织原则
- 模块间仅通过公开 API 通信
- 共享层不得承载业务逻辑
- `bootstrap` 必须作为集中装配层存在
- 当前范围是“最小可运行且边界清晰优先”
- 基础栈固定为 FastAPI、Pydantic、pytest、ruff
- 当前决策明确不接数据库

### 允许补全

- 为形成 `proposal/design/tasks` 所需的常规细化
- 在不改变边界前提下补充更具体的文件清单
- 细化启动链路说明、任务颗粒度与工程配置项
- 在模板范围内补充占位文件与说明文字

### 禁止脑补

- 引入真实业务模块
- 引入跨模块直接依赖内部实现的设计
- 将共享层扩张为通用业务容器
- 将数据库初始化、迁移链路、外部基础设施偷偷纳入当前 change
- 把未决事项伪装成已定方案

### 遇到未决事项的处理方式

如果后续文档生成阶段发现需要数据库、Docker 或其他基础设施，必须将其作为新的决策点显式提出；不能默认把它们并入当前 change。

## Suggested change naming

- `type`: `feature`
- `topic`: `project-scaffold-init`
- target path: `openspec/changes/feature-project-scaffold-init/`

该命名能稳定表达本次 change 的目标：初始化项目骨架，而不是补丁式修复或单纯研究。

# Pre-design: 应用代码最小 CI 实践

## 1. Problem framing

当前仓库已经具备平台中立的 CI/CD 基线与 `cicd-baseline` spec，用于定义从校验、构建到受控部署的治理边界；但这些内容仍然停留在通用治理层，尚未进一步收敛为“应用代码默认应经过哪些最小 CI 门禁、如何按流水线视角分层、哪些检查属于阻断项”的稳定前置设计。

因此，团队虽然已经具备 CI/CD 的总体边界认知，但还缺少一个专门面向应用代码、可以直接承接后续 OpenSpec proposal/design/tasks，并可稳定映射到具体流水线实现的中约束输入。这个缺口会导致后续文档或实现阶段容易出现两个问题：要么继续停留在抽象原则，无法指导落地；要么过早绑定某个 CI 平台，把平台实现误写成治理事实。

本次 pre_design 要解决的真实问题是：在保持平台中立和尊重当前仓库事实边界的前提下，为应用代码定义一套最小 CI 实践框架，使后续文档与实现能够明确回答“默认对哪些代码跑什么检查、最低分几层、哪些失败必须阻断、哪些内容不在本次 change 内”。

为什么现在做这件事：
- 仓库已经形成 `app/`、`tests/`、架构规则和测试分层规则，具备收敛应用代码 CI 实践的基础。
- 现有 `docs/cicd_baseline.md` 与 `openspec/specs/cicd-baseline/spec.md` 已经定义了通用 CI/CD 治理边界，当前最合理的下一步不是重写总基线，而是下沉出一个更接近流水线实现的应用代码实践层。
- 如果不先形成这个中间层，后续 proposal/design/tasks 很容易在“过于抽象”和“过早平台化”之间摇摆，难以稳定复用。

## 2. Goals / Non-goals

### Goals
- 定义一个面向应用代码的最小 CI 实践 change，作为现有平台中立 `cicd-baseline` 的下游实践约束。
- 明确默认覆盖对象以 `app/` 与其直接相关的 `tests/` 为主。
- 明确最小 CI 流水线至少包含哪些校验层与默认阻断门禁。
- 为后续 proposal/design/tasks 以及未来具体流水线实现提供稳定映射，但不直接绑定任何平台。
- 保持与现有架构规则、测试规则、`docs/cicd_baseline.md` 的阶段模型与术语体系一致。

### Non-goals
- 不在本 change 中重写完整 CI/CD 主基线。
- 不在本 change 中纳入文档类门禁或 docs/spec 一致性治理。
- 不在本 change 中扩展到受控部署、发布后验证、回滚入口或其他 CD 内容。
- 不生成 GitHub Actions、GitLab CI、Jenkins 或其他平台的具体配置文件。
- 不将镜像扫描、签名、审批链、多环境晋升或企业平台能力写成仓库现状。
- 不借此推进与目标无关的业务架构重构、测试体系重做或平台工程建设。

## 3. Requirement understanding

这次 change 的定位不是新增一篇通用 CI/CD 说明文档，也不是直接产出平台流水线配置，而是把“应用代码默认应经过哪些 CI 门禁”定义成一个足够清晰、边界收敛、且能直接承接后续流水线设计的中约束输入。

结合当前仓库状态，这个 change 应以以下事实为基础：
- 仓库已经具备应用代码骨架，如 `app/api.py`、`app/worker.py`、`app/bootstrap/api.py`、`app/interfaces/http/app.py`。
- 仓库已经具备测试分层骨架，如 `tests/unit/`、`tests/integration/`、`tests/architecture/`。
- 架构规则已明确强调 `service.py -> ports.py -> infrastructure/*` 调用链，以及依赖方向与边界控制。
- 测试规则已明确单元测试、集成测试、端到端测试、架构测试的职责分层。
- 现有 `docs/cicd_baseline.md` 已把默认 CI 门禁定义为交付准入条件的一部分，但本次需要进一步收窄到“应用代码最小实践”。

### 成功标准
后续基于本 pre_design 生成 OpenSpec 文档时，应能够稳定回答下列问题而无需再进行大幅脑补：
- 本 change 到底只覆盖哪些对象。
- 最小 CI 流水线至少分为哪些层次。
- 哪些检查属于默认阻断项。
- 哪些测试层级属于默认最小实践的一部分。
- 与现有 `cicd-baseline`、架构规则、测试规则分别如何对齐。
- 哪些内容明确不在本次范围内。

## 4. Constraints / Invariants

### Constraints
- 必须保持平台中立，不能把 GitHub Actions、GitLab CI、Jenkins 或其他平台语法写成默认前提。
- 必须与现有 `docs/cicd_baseline.md`、架构规则、测试规则保持术语与边界一致，不能另起一套治理模型。
- 必须面向后续流水线实现，因此不能只停留在抽象原则，需明确层次、门禁、适用范围与阻断语义。
- 只能聚焦应用代码最小 CI 实践，不吞并文档治理、CD、发布或回滚主题。
- 必须忠实于当前仓库事实边界，不把未确认已存在的工程资产或企业能力写成现状。

### Invariants
以下内容属于后续文档不得推翻的中约束：
- 本 change 只覆盖应用代码 CI，不覆盖 CD。
- 本 change 只定义最小流水线实践，不定义具体平台配置。
- 默认覆盖对象以 `app/` 及其直接相关测试为中心。
- 架构/边界检查是默认阻断项，不能降级为提示信息。
- 测试必须遵循现有测试分层，不允许以单一重型 E2E 替代分层校验。
- 任何后续文档都不得把企业平台能力、制品治理能力、多环境流程或未落地资产写成仓库现状。

## 5. Architecture direction

推荐把本 change 定义为一个面向后续流水线实现的“应用代码最小 CI 实践基线”。它位于现有平台中立 `cicd-baseline` 之下，负责将通用治理要求进一步收敛为应用代码可执行的默认门禁集合，但不取代现有主基线文档。

职责分层建议如下：
- `cicd-baseline`：定义通用 CI/CD 阶段模型与治理边界。
- 本次 change：定义应用代码默认应经过的最小 CI 实践。
- 后续具体实现：再把这些实践映射到某个平台的作业、阶段、触发条件与执行方式。

### 建议的最小 CI 分层
建议后续 design 采用三层最小流水线视角：

1. **静态与结构校验层**
   - 负责代码格式或静态检查、架构/边界检查，以及必要的依赖与安全最低检查。
   - 目标是尽早阻断结构性违规和明显不满足交付准入条件的变更。

2. **测试校验层**
   - 负责单元测试、已纳入默认准入的集成测试，以及架构测试。
   - 目标是验证业务编排、边界适配与关键结构约束在当前最小实践范围内可稳定通过。

3. **可运行性 / 构建确认层**
   - 负责必要构建校验、装配可导入或等价的运行时确认。
   - 目标不是生成完整可发布制品，而是确认变更满足进入后续交付阶段的最小可运行前提。

这个三层模型既保留了现有 `cicd_baseline` 的阶段语义，又能让后续 design 更自然地映射为平台无关的 stage/job 设计，而不直接滑向具体平台语法。

## 6. Key decisions / Trade-offs

### Key decisions
1. **本 change 是 `cicd-baseline` 的下游实践约束，而不是平级替代文档。**
   - 后续 proposal/design/tasks 应在 `cicd-baseline` 的治理边界内继续下沉，而不是重新定义整个 CI/CD 体系。

2. **默认覆盖对象以 `app/` 与其直接相关测试为中心。**
   - 本次 change 优先服务应用代码实践，而非文档门禁或完整工程治理。

3. **架构/边界检查必须是默认阻断项。**
   - 这与现有 `docs/cicd_baseline.md` 和架构规则一致，不能在后续设计中弱化为提示信息。

4. **最小 CI 以分层校验组织，而不是把所有检查混成单一命令列表。**
   - 后续 design 应明确不同校验层的目标、阻断关系与输入输出，而不是仅堆叠命令。

5. **保持平台中立，但允许后续 design 用 stage/job 语言描述映射思路。**
   - 可以描述平台无关的流水线结构，不可以写具体平台语法或平台专属功能。

### Trade-offs
#### 方案 A（推荐）：三层最小 CI 实践基线
- 内容：静态与结构校验层 / 测试校验层 / 可运行性与构建确认层。
- 优点：边界最稳，最容易承接后续 OpenSpec 设计与平台实现映射。
- 缺点：不直接覆盖文档门禁与 CD。

#### 方案 B：两层模型（检查层 + 测试层）
- 优点：表达更简洁。
- 缺点：会弱化“接近流水线”的结构表达，容易混淆构建确认职责。

#### 方案 C：端到端流水线骨架优先
- 优点：更贴近未来平台实现。
- 缺点：更容易过早引入平台细节，削弱平台中立性。

### Recommendation
选择方案 A。它最符合本次 change 的目标：在不直接绑定平台的前提下，把应用代码最小 CI 实践收敛成可稳定映射到后续流水线实现的中约束。

## 7. OpenSpec mapping

### `proposal.md` 应表达什么
- 为什么需要在现有 `cicd-baseline` 之下继续下沉出“应用代码最小 CI 实践”。
- 当前缺口：已有治理边界，但缺少面向应用代码、可稳定映射到流水线的最小实践定义。
- 本次 change 带来的改进：
  - 明确默认覆盖对象。
  - 明确最小 CI 分层。
  - 明确默认阻断门禁。
  - 明确与架构规则、测试规则、现有 CI/CD 基线的对齐关系。
- 明确本次不做什么：文档门禁、CD、平台配置、企业能力假设。

### `design.md` 应展开什么
- 三层最小 CI 结构化设计。
- 每层的目标、输入、输出、失败语义与阻断关系。
- 哪些检查属于默认必需，哪些属于后续扩展项。
- 如何与现有测试分层、架构规则和 `uv` 命令基线对齐。
- 如何保持平台中立同时支持后续映射到 stage/job。

### `tasks.md` 应拆解什么
- 文档与 spec 的补充或收敛任务。
- 最小 CI 实践所需检查项的收敛任务。
- 现有测试、架构测试、运行时校验与最小构建校验的对齐任务。
- 若后续需要落具体平台实现，应拆为独立实现任务，而不是在本 change 内一次吞并。

## 8. Generation guardrails

### 必须遵守
- change 主题是“应用代码最小 CI 实践”，不是完整 CI/CD 重写。
- 范围仅限 CI，不扩展到 CD。
- 范围仅限应用代码，不吞并文档治理。
- 架构/边界检查是默认阻断项。
- 测试应遵循现有测试分层，而不是退化成单层大杂烩。
- 保持平台中立，不写具体平台语法。
- 尊重当前仓库事实边界，不把未落地能力写成现成事实。

### 允许补全
- 进一步细化三层 CI 中各层的职责、顺序与阻断关系。
- 细化哪些现有测试或校验适合纳入默认最小门禁。
- 用更清晰的 stage/job 语言描述平台无关映射关系。
- 补充与 `uv`、导入校验、运行时装配校验相关的最小实践细节。

### 禁止补全
- 擅自将 scope 扩展到文档门禁、CD、发布、回滚、多环境晋升。
- 把 GitHub Actions、GitLab CI、Jenkins 写成默认实现。
- 把镜像扫描、签名、审批链、制品仓库准入等写成已存在能力。
- 把未决事项包装成既定事实。
- 借机引入与目标无关的架构重构或测试体系重做。

### 遇到未决事项时的处理原则
- 若未决事项影响 scope：优先收窄，不扩大。
- 若未决事项影响平台实现：保持平台中立表述，不提前绑定。
- 若未决事项影响默认门禁是否阻断：按“最小必需、可稳定执行、与现有规则一致”判断。
- 若某项能力当前仓库是否已存在不明确：按“不可写成现成事实”处理。

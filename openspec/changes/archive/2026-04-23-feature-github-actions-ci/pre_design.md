# pre_design：使用 GitHub Actions 落地最小可用 CI

## 1. Problem framing

### 要解决的问题
当前仓库已经具备平台中立的 CI/CD 基线与应用代码最小 CI 基线，但尚未在 GitHub Actions 上落地为可执行的 CI 资产。

这导致两类问题：
- 已定义的默认门禁缺少具体平台承载，无法直接在 GitHub 仓库协作中自动执行。
- 平台中立文档虽然给出了最小 CI 的范围、三层模型与推荐命令入口，但缺少将这些约束转化为实际仓库门禁的实现映射。

### 背景与触发原因
仓库内已有以下事实基础：
- `docs/cicd_baseline.md` 定义了平台中立的 CI/CD 主基线。
- `docs/application_minimal_ci_baseline.md` 定义了应用代码最小 CI 的默认覆盖范围、三层最小流水线模型与默认阻断门禁。
- `pyproject.toml` 已明确 `uv` 为依赖管理与命令执行基线。
- 当前仓库尚无 `.github/workflows/`，即尚未存在 GitHub Actions 层面的落地实现。

因此，本次 change 的核心不是重新设计 CI 规范，而是将现有基线稳定映射到 GitHub Actions。

### 当前方案/现状的核心不足
当前仓库的不足不在于缺少 CI 规范，而在于：
- 缺少 GitHub Actions 形式的自动执行入口。
- 缺少面向 `push` / `pull_request` 的最小代码侧阻断门禁。
- 缺少将“平台中立三层模型”映射为 GitHub Actions workflow/job 编排的固定实现方向。

### 为什么现在做
在已经形成平台中立基线的情况下，继续只停留在文档层会使默认门禁难以进入日常协作流程。尽快补齐 GitHub Actions 落地层，可以让已有基线从“规范”变为“可执行门禁”，为后续增量演进提供稳定起点。

## 2. Goals / Non-goals

### Goals
本次 change 要达成以下目标：
- 使用 GitHub Actions 落地最小可用 CI。
- 仅覆盖代码侧默认门禁，不扩展到 CD。
- 采用“单 workflow、多 job”的方式映射现有三层最小流水线模型。
- 保持与 `docs/application_minimal_ci_baseline.md` 中现有命令入口和职责边界一致。
- 使用 `uv` 作为命令执行基线。
- 让主分支与 Pull Request 的代码变更具备自动执行、自动阻断的最小 CI 能力。

### Non-goals
本次 change 明确不做以下事项：
- 不做部署。
- 不做完整 CD。
- 不把文档一致性校验纳入本次默认阻断门禁。
- 不引入镜像构建、制品上传、签名、扫描、审批链、多环境晋升等增强项。
- 不重新设计平台中立的 CI/CD 主基线。
- 不把未来可能需要的企业平台能力写成当前已落地事实。

### 看起来相关但本次不纳入的内容
以下事项可能在主题上相关，但不属于本次 change 的范围：
- `docs/` 或 OpenSpec 文档的一致性治理。
- 发布后验证、回滚入口、环境准入等 CD 边界内容。
- GitHub Actions 缓存、矩阵、多 Python 版本、多操作系统兼容等优化项。
- 针对镜像、供应链或企业安全平台的额外治理要求。

## 3. Requirement understanding

### 对需求的深层理解
本次需求不是“给仓库加一个 CI 文件”这么简单，而是要在不破坏既有平台中立表达的前提下，为 GitHub 仓库协作补一个最小但稳定的执行层。

因此，重点不在于使用多少 GitHub Actions 特性，而在于以下几点：
- 让 GitHub Actions 成为既有基线的实现映射层，而不是新的规则来源。
- 保持默认门禁最小、稳定、可执行。
- 保持后续扩展空间，但不在本次 change 内预支未来复杂度。

### 隐含约束
结合现有文档与仓库事实，可推导出以下隐含约束：
- GitHub Actions 只是平台承载方式，不能改写 `docs/cicd_baseline.md` 与 `docs/application_minimal_ci_baseline.md` 的职责边界。
- workflow 中的执行命令应尽量复用已有文档推荐入口，避免在平台配置里重新定义一套 CI 规则。
- 本次变更应优先围绕“最小必需、可稳定执行、与现有规则一致”来收敛。
- 由于采用单 workflow 多 job，必须显式说明哪些 job 可拆分，哪些阶段必须通过 `needs` 保持阻断顺序。

### 成功标准
本次 change 的成功标准是：
- 针对主分支与 Pull Request 的代码变更，可通过 GitHub Actions 自动运行最小代码侧 CI。
- 失败时能够作为默认阻断门禁使用。
- 落地方案与现有三层最小流水线模型一致。
- 不扩大到部署、文档一致性门禁或其他 CD 语义。

### 对业务/系统边界的解释
本次 change 处于“工程治理落地层”，不是业务架构变更。它应服务于仓库代码准入，不应触碰业务分层、领域边界或应用运行时装配逻辑本身。

## 4. Constraints / Invariants

### 必须遵守的约束
- GitHub Actions 只是平台映射层，不能改写既有 CI 基线的职责边界。
- 本次只覆盖代码侧最小 CI，不扩展到 CD。
- 默认门禁应围绕现有最小集合展开，而不是新增与当前目标无关的治理项。
- 执行入口应优先复用文档中已有命令语义，而不是在 workflow 里重新定义一套检查逻辑。
- `uv` 必须作为依赖与命令执行基线。

### 关键不变量
后续 OpenSpec 文档与实现阶段不得推翻以下不变量：
- 三层模型仍然成立：静态/结构 -> 测试 -> 可运行性确认。
- 架构/边界检查必须是默认阻断项，不能降级为提示。
- 文档一致性、部署、发布、回滚及企业平台增强能力不在本次 change 内。
- GitHub Actions 配置不得把未来能力写成当前仓库事实。

### 容易被误解的硬边界
- 这是“现有基线到 GitHub Actions 的映射”，不是“重新定义 CI 基线”。
- 这是“最小可用 CI”，不是“完整 CI/CD”。
- 这是“单 workflow 多 job 的最小实现方向”，不是“多 workflow 平台工程拆分”。
- 本次不以“覆盖最全”为目标，而以“最小、稳定、可阻断”为目标。

## 5. Architecture direction

### 推荐的总体架构方向
采用 **一个 GitHub Actions workflow + 多个 jobs** 的结构，把平台中立三层模型映射到 GitHub Actions 的 job 编排中。

建议方向如下：
- 以 `.github/workflows/ci.yml` 作为唯一入口。
- workflow 面向 `push` / `pull_request` 触发。
- jobs 按职责拆分，但整体仍服从三层最小流水线模型。
- 通过 `needs` 保持“前层失败不继续后层”的阻断语义。

### 关键模块边界
建议按稳定职责边界拆分 jobs，而不是按单个工具或命令过度切碎：

1. **static-and-structure**
   - 承接静态与结构校验层
   - 对应 `ruff check` 与架构/边界检查

2. **test-suite**
   - 承接默认测试集合中的单元测试
   - 可根据设计决定是否包含架构测试复跑，但不应改变其在第一层已作为阻断项的事实

3. **runtime-validation**
   - 承接集成测试 / 最小运行性确认
   - 用于表达“可运行性与构建确认层”的最小落地

### 职责划分原则
- workflow 负责组织执行与门禁顺序。
- 各 job 负责承接现有基线中已定义的执行职责。
- 基线规则仍然来自现有文档，而不是来自 GitHub Actions 配置本身。

### 哪些边界必须稳定
以下边界应在后续设计与实现中保持稳定：
- 单 workflow，不拆多个 workflow。
- 多 job，不退回到单 job 串行脚本。
- 三层模型语义不变。
- 不把 CD、文档治理或企业平台增强内容并入本次 CI change。

## 6. Key decisions / Trade-offs

### 已确认的关键决策
1. **采用单 workflow、多 job 方案**
   - 使用一个 `ci.yml` 作为最小代码侧 CI 的唯一入口。
   - 在同一个 workflow 内拆分多个 jobs，并通过 `needs` 表达阻断关系。

2. **以现有基线文档命令语义为准**
   - GitHub Actions workflow 是承载层，不重新发明 CI 规则。
   - 优先映射 `docs/application_minimal_ci_baseline.md` 中给出的推荐命令入口。

3. **本次只做最小代码侧 CI**
   - 不纳入文档一致性门禁。
   - 不纳入部署、发布、回滚或其他 CD 范围。

4. **不引入增强型平台能力**
   - 暂不引入缓存、矩阵、制品上传、镜像构建等扩展项。
   - 避免为了未来可扩展性而提前复杂化当前方案。

### 为什么这样选
- 单 workflow 多 job 更贴近 GitHub Actions 的常见使用方式，也为后续扩展保留自然演进路径。
- 相比单 job 串行脚本，这种方式在保留最小实现边界的同时，能更清楚表达阶段职责与阻断关系。
- 相比多个 workflow，这种方式更符合当前“最小可用 CI”的目标，不会在入口、触发与协作心智模型上增加额外复杂度。

### 明确放弃的替代方向
本次 change 明确不采用以下方向：
- **单 job 串行脚本**：虽然更简单，但不符合已确认的 GitHub Actions 方向选择。
- **多个 workflow 拆分**：会放大配置复杂度，并更容易让范围膨胀。
- **一步到位做完整 CI/CD**：会混入部署、发布与环境治理问题，超出本次边界。
- **把文档一致性一并纳入**：当前规则边界尚未收敛到可稳定平台化执行，不应在本次最小 change 中顺带解决。

## 7. Critical flows

### 核心控制流
本次 change 的核心控制流应表达为：
1. 代码变更触发 GitHub Actions workflow。
2. workflow 先进入静态与结构校验层。
3. 只有前层通过，才进入默认测试集合层。
4. 只有测试层通过，才进入最小可运行性确认层。
5. 任一阻断层失败，则整体 CI 失败。

### 关键数据流 / 配置流
- 仓库根目录的 `pyproject.toml` 提供 Python 依赖与 `uv` 执行基线。
- workflow 应以仓库已有测试目录与命令入口为主要执行对象。
- GitHub Actions 中的 job 编排只承接这些命令，不改变其语义。

### 重要异常路径
需要在后续设计中显式考虑的异常路径包括：
- `uv` 环境未准备完成导致命令无法执行。
- 架构测试或集成测试路径与当前文档中声明的资产存在偏差。
- job 拆分后因依赖关系配置错误导致三层语义失真。

这些异常路径应在 `design.md` 中被处理，但不应因此扩大本次 change 范围。

## 8. OpenSpec mapping

### `proposal.md` 应承接什么
后续 proposal 应重点表达：
- 为什么需要将现有平台中立最小 CI 基线映射到 GitHub Actions。
- 当前仓库的主要缺口是“已有基线、缺少 GitHub Actions 落地资产”。
- 本次 change 的边界是“最小代码侧 CI”，不含文档一致性与 CD。

### `design.md` 应承接什么
后续 design 应重点展开：
- `.github/workflows/ci.yml` 的整体结构。
- workflow 的触发条件。
- jobs 的职责划分与 `needs` 依赖关系。
- 如何将 `docs/application_minimal_ci_baseline.md` 中的命令入口映射到具体 jobs。
- GitHub Actions 常规实现细节，例如 Python/uv 环境准备方式。

### `tasks.md` 应承接什么
后续 tasks 应围绕“让最小 CI 可稳定执行”来拆解，例如：
- 新增 `.github/workflows/ci.yml`。
- 校准 workflow 所依赖的命令入口与现有测试资产。
- 验证 job 划分与三层阻断语义一致。
- 在必要时补充最小说明文档。

### 哪些内容不应由后续文档自行扩写
后续文档不得自行扩写以下内容：
- 部署、发布、回滚相关流程。
- 文档一致性门禁。
- 镜像构建、制品上传、矩阵、缓存等增强能力。
- 推翻单 workflow 多 job 的既定方向。

## 9. Generation guardrails

### 必须遵守
后续生成 proposal/design/tasks 时必须遵守：
- 本次 change 的主题是“GitHub Actions 落地最小代码侧 CI”。
- 架构方向固定为：**单 workflow、多 job**。
- 保留三层门禁语义：静态/结构 -> 测试 -> 可运行性确认。
- 以 `uv` 作为命令执行基线。
- 以现有基线文档的命令语义为主，不重新定义平台中立规则。

### 允许补全
后续文档允许补全的内容包括：
- 具体 job 名称。
- `needs` 的精细编排方式。
- `actions/checkout`、Python 安装、uv 安装等 GitHub Actions 常规实现细节。
- 为了让 workflow 可执行而进行的最小工程化补充。

### 禁止补全
后续文档禁止擅自补全以下内容：
- 擅自扩大到部署、发布、回滚。
- 擅自加入文档一致性门禁。
- 擅自加入缓存、矩阵、多平台、多版本、制品上传、镜像构建等增强项。
- 把未来可能需要的 CI/CD 能力写成这次 change 的目标。
- 把未决事项伪装成已决定事项。

### 遇到未决事项时的处理原则
如果后续生成 proposal/design/tasks 时遇到未决事项，应遵循以下原则：
- 可以补充不改变核心架构意图的常规实现细节。
- 不能通过脑补扩大范围或改变边界。
- 如果某事项会影响“单 workflow、多 job、最小代码侧 CI”的核心判断，应显式标记为待决，而不是擅自定案。

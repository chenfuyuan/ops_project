# 项目初始化前置设计

## 1. Problem framing

当前仓库已经具备一套较完整的架构基线文档与 OpenSpec 配置，但仍缺少一个可直接承接后续 OpenSpec 生成的“项目初始化前置设计”。如果后续直接从零生成 `proposal.md`、`design.md` 或 `tasks.md`，很容易出现范围漂移：有人会把它理解成“先搭几个目录”，有人会把它扩成“直接实现项目骨架”，还有人会忽略治理、依赖边界与演进约束，只保留表面结构。

因此，本 change 的核心任务不是重复架构说明，而是将 `docs/` 中已经明确的架构基线、技术基线和治理要求，沉淀为一个可稳定传递给后续 OpenSpec 产物的中约束输入。该输入需要清楚界定：

1. 新项目初始化到底要初始化什么；
2. 这次初始化明确不做什么；
3. 后续文档哪些内容可以细化，哪些内容不得擅自改写。

本次前置设计服务的是“新项目骨架初始化”这一件事，而不是具体业务开发，也不是直接实现工程代码。

---

## 2. Goals / Non-goals

### 2.1 Goals

本 change 需要达成以下目标：

- 定义“新项目初始化”这一 change 的目标与边界；
- 固化项目骨架的顶层结构、运行形态与技术基线；
- 固化与项目初始化直接相关的治理护栏，包括依赖方向、准入规则、测试与评审重点；
- 为后续 OpenSpec 文档生成提供不可随意推翻的中约束。

### 2.2 Non-goals

本 change 明确不包含以下内容：

- 不直接生成实现代码；
- 不直接产出 `proposal.md`、`design.md`、`tasks.md`；
- 不引入具体业务域、业务流程或领域规则设计；
- 不展开 CI/CD、云资源编排、详细运维 SOP；
- 不把“未来可能的微服务拆分”扩展成当前 change 的实现范围。

换言之，这次 change 的目标是为“初始化项目”建立稳定设计边界，而不是把整个项目一次性设计到实现级别。

---

## 3. Requirement understanding

这次 change 的本质不是“整理一份架构摘要”，而是把 `docs/` 下的架构基线转化为一个可以驱动后续 OpenSpec 生成的初始化约束。它至少需要回答以下三个问题：

1. 新项目默认骨架长什么样；
2. 项目启动时必须同时建立哪些边界与治理规则；
3. 后续文档在哪些地方允许细化，在哪些地方不能偏离当前基线。

从现有文档来看，该初始化方案不是一个空白自由设计题，而是一个带有明确事实来源和强约束的基线承接任务：

- `docs/architecture_baseline/` 已经定义了顶层模块边界、业务内部结构、防腐方式、依赖方向与治理重点；
- `docs/technical_architecture_baseline.md` 已经定义了默认运行时形态、技术栈与基础设施落位原则；
- `openspec/config.yaml` 已经约束所有 OpenSpec 产物必须使用中文编写。

因此，`pre_design` 的职责不是重新发明架构，而是把这些上游事实转化为一个可被后续 `proposal/design/tasks` 稳定继承的前置设计输入。

---

## 4. Constraints / Invariants

### 4.1 Constraints

本次前置设计必须遵守以下约束：

- 必须以 `docs/architecture_baseline/*` 和 `docs/technical_architecture_baseline.md` 作为上游事实来源；
- 必须保持“模块化单体 + API / Worker 双进程 + PostgreSQL / Redis / S3-compatible + FastAPI / SQLAlchemy / Alembic / Celery + Python 3.13 + uv 依赖管理”的默认技术基线；
- 必须坚持 `business / capabilities / interfaces / shared / bootstrap` 作为顶层边界；
- 必须把治理要求纳入初始化范围，而不只写目录结构；
- 所有 OpenSpec 产物必须使用中文。

### 4.2 Invariants

以下内容属于后续文档不得推翻的不变量：

- `business` 是唯一承载业务语义的主区域；
- `capabilities` 只承载无业务语义的通用能力；
- 业务调用通用能力必须遵循 `service -> ports -> infrastructure` 的边界链路；
- `shared` 不能成为杂物间或跨业务耦合通道；
- 运行时实现选择必须集中在 `bootstrap` 或边界适配区，而不能回流到业务代码；
- 本 change 只定义项目初始化骨架，不定义任何具体业务实现。

这些约束不是建议项，而是后续 OpenSpec 文档必须继承的中约束。

---

## 5. Architecture direction

本 change 采用“骨架优先、治理内置”的初始化方向：先定义一个可稳定落地的模块化单体骨架，再把后续最容易腐化的边界一起前置固化。

这意味着项目初始化不是“先搭目录，治理以后再补”，而是将以下事项视为同一个初始化问题的不同侧面：

- 顶层模块与节点边界；
- 默认运行时拓扑；
- 基础设施职责与数据落位；
- 依赖方向与防腐方式；
- 测试、评审与准入治理。

该方向最适合当前仓库状态：仓库仍处于基线文档阶段，尚未进入具体业务开发，因此最需要的是一个“可稳定承接后续生成”的初始化 change，而不是立即把讨论推进到实现级细节。

---

## 6. Key decisions / Trade-offs

### 6.1 已确认的关键决策

- 以**模块化单体**作为默认架构形态，而不是预设微服务拆分；
- 以 **API 进程 + Worker 进程** 作为默认运行时拓扑；
- 以 **PostgreSQL / Redis / S3-compatible Object Storage** 作为默认数据与基础设施落点；
- 以 **FastAPI / Pydantic v2 / SQLAlchemy 2.x / Alembic / Celery / Python 3.13 / uv** 作为技术基线；
- 以 `business / capabilities / interfaces / shared / bootstrap` 作为顶层固定边界；
- 以 `ports + infrastructure` 作为业务防腐的强制模式；
- 将**架构治理**纳入初始化范围，包括 import 约束、架构测试、高敏感评审区域、命名规则与测试分层。

### 6.2 关键取舍

#### 为什么不采用“先最小目录、后续再补规则”

这种方式短期更快，但会在项目最早期就形成错误依赖、`shared` 污染和能力层越界；等到业务代码增多后再补规则，成本显著提高。

#### 为什么不采用“直接面向未来微服务设计”

虽然这种方式强调长期演进，但对当前初始化来说会过早放大范围，把问题从“如何落一个稳定单体骨架”扩展成“如何设计未来服务化路线图”，不利于形成清晰 change 边界。

#### 为什么要把治理写进初始化

因为这套基线的核心价值不是目录长什么样，而是边界是否可执行。如果没有 import 约束、架构测试、命名规范和高敏区评审机制，骨架很快会偏离最初设计意图。

---

## 7. 推荐初始化骨架范围

后续文档在展开初始化方案时，应围绕以下骨架范围组织，而不是脱离这些边界自由发散：

### 7.1 顶层结构

```text
project_scaffold/
├─ app/
│  ├─ business/
│  ├─ capabilities/
│  ├─ interfaces/
│  ├─ shared/
│  └─ bootstrap/
```

### 7.2 运行时形态

默认采用：

- API 进程：承载 HTTP 接口、同步业务入口与异步任务投递；
- Worker 进程：承载耗时任务、可重试任务、文件处理与后台作业；
- PostgreSQL：承载核心业务数据与持久化元数据；
- Redis：承载缓存、轻量协调与异步任务基础支撑；
- S3-compatible Object Storage：承载文件、大对象与产物。

### 7.3 初始化必须同步纳入的治理项

后续初始化设计不能只描述目录和进程，还应同时覆盖：

- import 规则与依赖方向约束；
- `service.py` 不得直接依赖 SDK / ORM / HTTP client 的架构测试；
- `shared/`、`capabilities/`、`workflow/state.py`、`business/**/ports.py`、`business/**/infrastructure/*` 的高敏感评审策略；
- 端口命名、基础设施实现命名、目录命名的统一规范；
- 单元测试 / 集成测试 / 端到端测试 / 架构测试的分层策略。

---

## 8. OpenSpec mapping

### 8.1 `pre_design.md` 应承接的内容

`pre_design.md` 负责定义项目初始化的目标、边界、架构方向、关键约束与治理护栏，作为后续文档的中约束输入。

### 8.2 后续 `proposal.md` 应承接的内容

后续 `proposal.md` 应聚焦以下内容：

- 为什么要做这个初始化 change；
- 该 change 将建立哪些骨架能力与边界；
- 影响范围和预期收益是什么。

`proposal.md` 不应重新解释整套架构原理，也不应改写当前基线中的关键边界定义。

### 8.3 后续 `design.md` 应承接的内容

后续 `design.md` 应重点展开：

- 目录组织如何具体落地；
- API / Worker / bootstrap 的入口与装配如何设计；
- 配置、数据库迁移、缓存、异步任务、对象存储如何建立基础接入点；
- 架构治理机制如何通过测试、检查或约定落盘。

### 8.4 后续 `tasks.md` 应承接的内容

后续 `tasks.md` 应拆解为可执行的初始化任务，例如：

- 目录骨架创建；
- 应用入口与启动装配；
- 配置体系建立；
- 数据库迁移基座与数据访问基座；
- Celery 与 Redis 接线；
- S3 适配基座；
- 架构测试与依赖准入规则接入。

---

## 9. Generation guardrails

### 9.1 后续文档必须遵守

后续 OpenSpec 文档必须遵守以下规则：

- 模块化单体是当前默认架构形态；
- `business / capabilities / interfaces / shared / bootstrap` 是固定顶层边界；
- `ports + infrastructure` 是业务边界防腐的标准模式；
- 治理要求属于初始化范围，不得在后续文档中弱化为“可选项”；
- 不得把本 change 擅自扩展为具体业务域实现或生产级平台建设。

### 9.2 后续文档允许补全

后续文档允许补全以下内容：

- 更细的目录层级与骨架文件组织；
- 配置、启动、测试、依赖检查的具体落地方式；
- 不改变核心边界的实现细化。

### 9.3 后续文档禁止脑补

后续文档禁止擅自补全以下内容：

- 新业务目标；
- 与当前基线冲突的技术栈替换；
- 未经确认的微服务拆分方案；
- 超出初始化范围的 CI/CD、云资源、运维体系建设。

如果后续生成过程中遇到未决事项，应将其明确标记为待确认问题，而不是伪装成已决定事实。

---

## 10. 结论

本次 `feature-project-bootstrap-baseline` change 的定位，是把现有文档中已经定义的新项目架构基线、技术基线与治理规则，整合为一个稳定的“项目初始化前置设计”。它的价值不在于补充实现细节，而在于为后续 OpenSpec 产物建立清晰边界、显式取舍与不可漂移的设计护栏。

后续任何围绕项目初始化的 OpenSpec 生成，都应以本文件为上游中约束输入，而不应重新定义项目初始化的目标、顶层边界或治理方向。

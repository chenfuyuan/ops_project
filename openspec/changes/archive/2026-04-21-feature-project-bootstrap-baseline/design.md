## Context

当前仓库处于“有架构基线文档、无实现骨架”的初始阶段。`docs/architecture_baseline/*` 已定义顶层模块边界、节点内部结构、防腐方式与治理重点，`docs/technical_architecture_baseline.md` 已定义默认运行时拓扑、技术栈和基础设施职责，但这些内容尚未被整理为一个可直接指导实现的初始化设计。

本 change 的上游约束来自 `pre_design.md`，其已经明确：

- 当前目标是初始化新项目骨架，而不是进入具体业务实现；
- 默认架构形态是模块化单体，而不是预设微服务；
- 必须同时建立顶层边界、运行时拓扑、基础设施基线和治理护栏；
- 后续设计只能在这些约束内展开，不能擅自扩展到 CI/CD、云资源或业务域设计。

当前设计面对的是一个典型的“从文档共识走向工程骨架”的问题：需要把架构意图转化为可落地的项目结构、进程入口、装配方式和基础设施接入点，同时保证后续代码不会在初始阶段就破坏边界。

## Goals / Non-Goals

**Goals:**
- 定义项目初始化骨架的落地方式，使其与现有架构基线一致。
- 明确 `app/` 顶层结构及各层在初始化阶段的职责边界。
- 定义 `API 进程 + Worker 进程` 的默认入口与装配方式。
- 定义 PostgreSQL、Redis、S3-compatible Object Storage 在初始化阶段的基础接入点。
- 定义配置体系、数据库迁移基座、异步任务基座和共享基础设施的最小落地方案。
- 定义 `Python 3.13 + uv + pyproject.toml + uv.lock` 的项目依赖与环境管理基线。
- 定义初始化阶段就应接入的治理机制，包括 import 约束、架构测试、命名规范和高敏感区域评审关注点。

**Non-Goals:**
- 不设计具体业务域、业务节点和业务流程。
- 不设计生产环境部署、CI/CD 流水线和云资源拓扑。
- 不引入超出基线的新技术栈替换方案。
- 不把当前初始化设计扩展为未来微服务拆分的详细迁移方案。

## Decisions

### 1. 以单一应用仓库承载初始化骨架

**决策**：初始化阶段使用单一应用仓库结构，在 `app/` 下固定建立 `business / capabilities / interfaces / shared / bootstrap` 五个顶层区域，并以此作为后续所有实现的准入边界。

**原因**：当前阶段的目标是把架构边界先固定下来，而不是根据尚不存在的业务复杂度做过度分拆。单仓库、单应用骨架能够最快承载后续实现，同时保留足够清晰的边界。

**备选方案**：
- 直接预留多服务目录：长期想象空间更大，但会把初始化范围拉向微服务设计。
- 只建立技术分层目录：落地更快，但会弱化业务边界表达，违背基线中的“业务优先，技术从属”。

### 2. 采用 API 与 Worker 分离的双进程入口

**决策**：在初始化骨架中明确建立 API 入口与 Worker 入口，并通过 `bootstrap` 统一完成依赖装配、配置加载与实现选择。

**原因**：技术基线已经明确同步请求与后台任务是默认并存的运行形态。若初始化阶段只考虑 API，会导致后续接入异步任务时重新调整装配边界。

**备选方案**：
- 先只保留 API 入口：短期更省事，但会推迟 Worker 相关边界设计。
- 将装配逻辑散落到 `interfaces` 或任务定义中：实现方便，但会破坏“实现选择集中在 bootstrap”的约束。

### 3. 以 shared 提供系统级基础设施基座，以 bootstrap 连接运行时实现

**决策**：数据库连接、配置加载、日志/追踪封装、HTTP client 封装、对象存储客户端封装等系统级基础设施，优先放在 `shared/infra`；具体运行时装配、环境差异和实现选择统一放在 `bootstrap`。

**原因**：这样能够把“可复用基础设施”和“运行时选择逻辑”分开，避免 `shared` 反向承担装配职责，也避免 `bootstrap` 吞掉通用基础设施实现。

**备选方案**：
- 将所有基础设施都堆到 `bootstrap`：会让 `bootstrap` 变成实现杂糅层。
- 将所有客户端直接放到业务或能力层：会造成重复接线，并弱化系统级一致性。

### 4. 以骨架级接入点承载 PostgreSQL、Redis、S3 与 Alembic、Celery

**决策**：初始化阶段只建立这些基础设施的基座接入点与约束位置，不引入具体业务表结构或业务任务；数据库迁移通过 Alembic 建立空基座，异步任务通过 Celery 建立 Worker 应用与任务注册基座，对象存储建立统一客户端封装与引用传递约束。

**原因**：这既满足“现在就有可用工程骨架”的目标，也避免把项目初始化扩展为具体业务建模。

**备选方案**：
- 直接预置示例业务模型和任务：更直观，但会污染“无业务逻辑项目骨架”的边界。
- 完全不接这些基础设施，只留文档说明：会使后续实现缺少真实承接点。

### 5. 以 uv 固化 Python 依赖、锁文件与命令执行基线

**决策**：项目初始化阶段即固定 `Python 3.13 + uv + pyproject.toml + uv.lock` 作为依赖与环境管理基线，统一通过 `uv sync` 同步环境、通过 `uv run` 执行项目命令，并在初始化阶段仅维护 `runtime / dev / test` 三类依赖分组。

**原因**：当前项目属于工程化后端应用，而不是本地科学计算环境。将依赖定义、锁文件、虚拟环境与命令执行统一到 `uv` 体系，能够在 API / Worker 双进程骨架下保持本地、CI 与后续部署的一致性，同时避免并行维护多套依赖真相来源。

**备选方案**：
- 使用 `requirements.txt` + `venv`：更传统，但依赖分组与锁版本治理较弱。
- 使用 Conda 作为项目级依赖管理：适合重二进制或科学计算环境，但对当前后端骨架偏重。
- 同时维护多套依赖定义：短期兼容性更高，但会快速引入环境漂移与治理混乱。

### 6. 治理机制在初始化阶段即落盘

**决策**：初始化设计中必须包含可执行治理方案，至少覆盖 import 方向约束、架构测试、命名规范、测试分层和高敏感区域评审要求。

**原因**：pre_design 已明确治理属于初始化范围，而不是后续可选项。如果只实现目录不实现治理，结构很快会偏离基线。

**备选方案**：
- 仅靠文档约束：成本低，但执行力弱。
- 等业务代码出现后再补治理：更贴近真实代码，但通常已错过最便宜的约束时机。

### 6. 规格以“项目骨架能力”而非“业务能力”建模

**决策**：本次 specs 应围绕 `project-bootstrap-baseline` 这个项目初始化能力展开，要求覆盖结构基线、运行时基线、基础设施接入基线与治理基线。

**原因**：当前 change 没有业务域目标，所有需求都服务于“把项目初始化好”。以单一初始化能力建模最符合 proposal 和 pre_design 的边界。

**备选方案**：
- 将其拆成多个过细的技术 specs：可读性更细，但对当前仓库会增加碎片化管理成本。

## Risks / Trade-offs

- [风险：初始化骨架过重] → 通过“只建立基座、不填充业务实现”控制范围，避免把骨架阶段做成平台建设。
- [风险：shared 被过度使用] → 在设计中明确 shared 的准入标准，并用架构测试和评审守护。
- [风险：Worker 入口先建但短期未用] → 保留最小可运行接线，避免后续重新设计运行时边界。
- [风险：治理约束停留在文字层] → 在任务阶段要求落地为自动化检查、测试骨架或显式准入规则。
- [风险：后续实现擅自扩展范围] → 通过 specs 和 tasks 明确非目标，禁止引入业务域、CI/CD 或微服务拆分工作。

## Migration Plan

- 第一步：建立目录骨架与运行时入口，使项目具备 API、Worker、bootstrap 和 shared 基础设施承接点。
- 第二步：建立配置体系、数据库迁移基座、Redis/Celery 接线与对象存储基础封装。
- 第三步：建立 interfaces、capabilities、business 的模板化目录与占位模块，但不引入具体业务语义。
- 第四步：接入治理机制，包括 import 约束、架构测试、测试目录骨架和命名规范说明。
- 第五步：验证本地最小运行链路可用，并作为后续具体业务实现的基础。

回滚策略：如果初始化实现过程中发现某一基础设施接入方案与基线冲突，应回退到“仅保留接口与装配边界、不保留具体接线细节”的最小骨架状态，而不回退顶层边界和治理方向。

## Why

当前仓库已经具备架构、部署环境、安全、发布与回滚等工程治理文档，但缺少一个平台中立的 CI/CD 基线，无法作为后续流水线设计、文档补充与交付评审的统一约束入口。

如果没有这个 change，后续很容易直接跳到某个平台实现，导致以下问题：
- 质量门禁定义不一致
- 可发布制品的定义不清晰
- 部署准入、发布后验证与回滚入口边界不统一
- 新增 CI/CD 说明与现有发布、安全、部署文档职责重叠或冲突

现在推进这个 change，是为了先建立“平台中立、偏严格治理、到受控部署为止”的 CI/CD 中枢性规范，使后续 OpenSpec 设计与实现都在同一组边界和护栏下展开。

## What Changes

- 新增一项平台中立的 CI/CD 基线变更，用于定义项目默认的 CI/CD 阶段模型与职责边界。
- 定义默认质量门禁，覆盖格式/静态检查、测试、构建校验、文档一致性校验、架构/边界检查以及依赖/安全最低门禁。
- 定义“可发布制品”作为交付对象抽象，要求制品可追踪、可验证并可与发布和回滚关联，但不强绑定单一平台或唯一制品形式。
- 定义到“受控部署”为止的 CD 边界，包括部署触发、环境准入、发布后最小验证与回滚入口。
- 新增一篇 CI/CD 主文档，并同步约束后续对发布、安全、部署相关基线文档的定向补充，明确各自职责边界。
- 约束后续工作不扩展到具体平台语法、多环境晋升编排、云资源拓扑、运维 SOP 或企业平台能力假设。

## Capabilities

### New Capabilities
- `cicd-baseline`: 定义项目的 CI/CD 阶段模型、默认质量门禁、可发布制品抽象、受控部署边界，以及与发布/安全/部署文档的职责映射。

### Modified Capabilities
- 无

## Impact

- 影响 `openspec/changes/docs-cicd-baseline/` 下后续 spec、design、tasks 的生成方向。
- 影响后续新增的 CI/CD 主文档及其与 `docs/release_change_and_rollback.md`、`docs/security_baseline.md`、`docs/deployment_environment_baseline.md` 的职责分层。
- 影响未来具体流水线实现时的准入标准、检查项分层、交付物抽象与部署边界表达方式。
- 不直接引入任何具体 CI/CD 平台实现，不影响现有运行时代码或已上线系统行为。

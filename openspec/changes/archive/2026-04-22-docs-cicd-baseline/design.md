## Context

当前仓库已经具备架构、部署环境、安全、发布与回滚等基础治理文档，但尚未形成一个平台中立的 CI/CD 中枢性规范。已有 `pre_design.md` 已经明确：本次 change 的核心目标不是落地某个平台流水线，而是建立一个“平台中立、偏严格治理、到受控部署为止”的 CI/CD 基线，为后续文档与实现提供稳定护栏。

本 change 的约束包括：
- 不绑定具体 CI/CD 平台；
- 默认质量门禁必须包含架构/边界检查；
- 交付对象采用“可发布制品”抽象，而不是唯一镜像前提；
- CD 范围只到受控部署，不进入多环境晋升与复杂发布策略；
- 新增一篇 CI/CD 主文档，并与发布、安全、部署专题文档分层协作；
- 不得把未落地的平台能力或工程资产描述为现成事实。

当前设计要解决的问题，是如何在不绑定平台的情况下，把 CI/CD 阶段模型、默认门禁、交付物抽象、受控部署边界以及文档职责映射组织成一个可指导后续实现与文档补充的方案。

## Goals / Non-Goals

**Goals:**
- 定义平台中立的 CI/CD 阶段模型与职责边界。
- 定义偏严格治理的默认 CI 门禁分层。
- 定义“可发布制品”在交付闭环中的角色与约束。
- 定义受控部署的最小闭环，包括触发、准入、发布后验证与回滚入口。
- 定义新增 CI/CD 主文档与现有发布、安全、部署文档的职责映射。
- 为后续实现流水线或补充文档提供可映射、可审查的设计护栏。

**Non-Goals:**
- 不设计具体平台的流水线语法或目录结构。
- 不设计多环境晋升编排、复杂灰度或云资源拓扑。
- 不设计值班 Runbook、平台操作手册或组织流程。
- 不假定镜像仓库、签名、扫描、审批链等企业平台能力已经存在。
- 不直接进入实现代码或平台接线细节。

## Decisions

### 1. 采用平台中立的阶段模型，而不是平台导向设计

**决策**：设计以“校验 / 构建 / 可发布制品 / 受控部署 / 发布后验证 / 回滚入口”的平台中立阶段模型为核心，不在设计中引入 GitHub Actions、GitLab CI、Jenkins 等具体平台概念。

**原因**：pre_design 已将平台中立设为硬约束。先固定能力模型与阶段边界，后续才能在不改变治理意图的前提下映射到任意平台。

**备选方案**：
- 直接按 GitHub Actions 等平台结构来组织设计：实现感更强，但会提前绑定平台。
- 完全只写高层口号，不定义阶段边界：保持抽象，但不足以指导后续文档和实现。

### 2. 以默认门禁作为交付准入，而不是仅把测试视为 CI 核心

**决策**：将默认 CI 门禁定义为交付准入条件，至少包含格式/静态检查、测试、构建校验、文档一致性校验、架构/边界检查与依赖/安全最低门禁，并明确架构/边界检查为默认阻断项。

**原因**：当前仓库治理导向不只是“代码能跑”，还强调架构边界、安全与文档一致性。如果不把这些纳入默认门禁，CI/CD 将无法承接仓库已有治理基线。

**备选方案**：
- 只保留格式、测试和构建：范围更小，但无法体现现有治理重点。
- 把所有安全与治理能力都拉成强制重门禁：可能超出当前仓库已确认能力边界。

### 3. 以可发布制品作为交付闭环中心对象

**决策**：将“可发布制品”定义为交付闭环中的核心对象，要求其可追踪、可验证、可与发布和回滚关联。容器镜像作为优先兼容形式之一，但不是唯一合法形式。

**原因**：如果没有稳定的交付对象抽象，CD 将只能停留在“部署某个提交”的模糊状态，不利于定义准入、验证与回滚。与此同时，将镜像设为唯一前提会违背平台中立和当前仓库事实边界。

**备选方案**：
- 完全不定义交付对象：会使 CD 闭环失去锚点。
- 把容器镜像写成唯一制品：更具体，但会过早绑定实现前提。

### 4. CD 范围止于受控部署闭环

**决策**：将 CD 范围限定为部署触发、目标环境准入、发布后最小验证与回滚入口，不扩展到多环境晋升、审批编排、灰度/蓝绿/金丝雀或云平台交付拓扑。

**原因**：这与 pre_design 中确认的边界一致，既能形成真正的交付闭环，又避免将 change 范围膨胀到平台化运维设计。

**备选方案**：
- 只做到“可发布为止”：范围更稳，但 CD 价值不足。
- 继续展开到多环境晋升和复杂策略：闭环更完整，但显著超出本次边界。

### 5. 采用“CI/CD 主文档 + 专题文档联动”的文档架构

**决策**：新增一篇 CI/CD 主文档，负责承接阶段模型、默认门禁、可发布制品、受控部署边界与职责映射；现有发布、安全、部署文档继续各自承担专题基线，通过定向补充与交叉引用协作。

**原因**：如果只在现有文档中零散补充，无法形成统一入口；如果由 CI/CD 主文档重写所有相关内容，又会破坏既有文档职责边界。

**备选方案**：
- 单文档聚合：入口更集中，但重复度高。
- 只做散点补丁：改动更少，但缺乏中枢。

## Risks / Trade-offs

- [风险：平台中立导致设计过虚] → 通过明确阶段模型、门禁分层、制品抽象和部署闭环来保持可落地性。
- [风险：默认门禁过重] → 将门禁控制在当前仓库已明确的治理基线内，不引入未确认能力。
- [风险：CI/CD 主文档与专题文档重复] → 通过职责映射明确“中枢定义”和“专题基线”的分工。
- [风险：后续文档或实现擅自扩展范围] → 在 spec 与 tasks 中显式约束非目标，禁止引入平台绑定、多环境晋升和运维平台设计。
- [风险：把未来企业能力写成现状] → 在设计中反复强调未落地能力不得写成现成事实。

## Migration Plan

- 第一步：新增 CI/CD 主文档，定义阶段模型、默认门禁、可发布制品、受控部署边界与职责映射。
- 第二步：对 `docs/release_change_and_rollback.md` 做定向补充，使其与 CI/CD 交付闭环衔接。
- 第三步：对 `docs/security_baseline.md` 做定向补充，使其覆盖流水线与制品治理的最低安全要求。
- 第四步：对 `docs/deployment_environment_baseline.md` 做定向补充，使其表达部署触发与环境准入边界。
- 第五步：如后续进入实现阶段，再将默认门禁和阶段模型映射到具体平台流水线资产。

回滚策略：如果后续发现某一部分设计过度接近特定平台或引入未确认能力，应回退到“只保留平台中立阶段模型和治理约束”的最小表达，而不回退已确认的交付闭环边界。

## Open Questions

- 文档一致性校验在后续实现阶段具体如何自动化，需结合仓库实际工具链再定。
- 架构/边界检查最终采用何种自动化手段落地，需在实现阶段结合项目测试和检查工具选择。
- 若未来确认具体 CI/CD 平台，哪些能力映射属于平台适配层、哪些仍应保留为平台中立文档约束，需要在后续实现或平台化 change 中细化。

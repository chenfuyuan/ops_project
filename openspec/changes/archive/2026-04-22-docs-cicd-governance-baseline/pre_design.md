# pre_design: CI/CD governance baseline

## 1. Problem framing

### 1.1 Background
当前仓库已经形成一组较完整的工程治理文档，覆盖架构边界、安全、部署环境、发布与回滚等主题，但仍缺少一个用于统一定义 CI/CD 闭环的中枢性文档。

现状风险主要在于：
- 缺少平台中立的 CI/CD 主基线，后续实现和评审缺少统一约束入口。
- 现有发布、安全、部署文档虽与交付过程密切相关，但并未从“CI → 可发布制品 → 受控部署”这一闭环视角统一组织。
- 当前仓库不能被视为已经存在某种 CI/CD 平台实现，因此不能把 GitHub Actions、GitLab CI、Jenkins 或其他平台能力写成既成事实。
- 如果继续用分散文档承载 CI/CD 约束，容易出现质量门禁口径不一致、可发布制品定义不清、部署准入边界模糊、发布后验证不完整、回滚入口不统一等问题。

### 1.2 Why now
在已有治理文档体系逐步完善的前提下，下一步不应直接跳到某个平台实现，而应先补齐平台中立的 CI/CD 治理基线。只有先定义清楚“什么必须检查、什么才算可发布、在什么条件下进入部署、部署后最少验证什么”，后续 OpenSpec 文档和未来工程落地才不会发生范围漂移。

### 1.3 Core problem
本次 change 要解决的核心问题不是“补一条具体流水线”，而是建立一个可稳定传递给后续 OpenSpec 文档的 CI/CD 中约束输入：
- 定义 CI/CD 应覆盖哪些阶段。
- 明确默认质量门禁。
- 定义“可发布制品”这一交付对象抽象。
- 明确受控部署、最小发布后验证与回滚入口的边界。
- 明确 CI/CD 主文档与现有发布、安全、部署文档之间的职责分工。

## 2. Goals / Non-goals

### 2.1 Goals
本次 change 的目标是形成一个“平台中立、严格治理、范围到受控部署为止”的 CI/CD 前置设计输入，供后续 `proposal.md`、`design.md`、`tasks.md` 生成时遵守。至少应达成以下目标：

1. 定义统一的 CI/CD 阶段模型。
2. 定义默认 CI 门禁，并按严格治理处理，至少覆盖：
   - 格式或静态检查
   - 测试
   - 必要的构建校验
   - 文档一致性校验
   - 架构/边界检查
   - 依赖/安全最低门禁
3. 定义“可发布制品”作为交付对象抽象，不强绑定单一平台或单一制品形式。
4. 定义受控部署边界，包括：
   - 部署触发条件
   - 目标环境准入条件
   - 最小发布后验证
   - 回滚入口
5. 明确新增一篇 CI/CD 主文档，并与现有发布、安全、部署文档建立职责分层与引用关系。
6. 为后续 OpenSpec 文档生成提供稳定护栏，防止范围漂移到平台实现、多环境晋升或完整运维体系设计。

### 2.2 Non-goals
本次 change 明确不做以下事项：
- 不绑定任何具体 CI/CD 平台。
- 不生成 GitHub Actions YAML、Jenkinsfile、GitLab CI 配置或其他平台专属语法。
- 不展开多环境晋升、复杂灰度、蓝绿或金丝雀策略。
- 不展开云资源拓扑、运维 SOP、值班 Runbook 或平台操作手册。
- 不把镜像仓库、签名、扫描、审批链等企业平台能力写成已落地事实。
- 不把本 change 扩张成全面工程治理体系重构。
- 不进入实现代码编写，也不自动推进后续 OpenSpec artifact 生成。

## 3. Requirement understanding

### 3.1 Deep understanding
这次需求的核心不是“尽快接一个 CI/CD 工具”，而是先建立一个能够约束后续实现的 CI/CD 治理基线。重点在于回答以下问题：
- 哪些检查属于默认准入门槛。
- 什么算可发布交付物。
- 什么条件下可以进入部署。
- 部署后至少要验证什么。
- 回滚入口如何定义。
- 这些要求应分别落在什么文档里，避免职责重叠和遗漏。

### 3.2 Repo-state assumptions to preserve
基于当前代码库与文档状态，需要保留以下事实边界：
- 当前仓库不能被表述为“已经有某个具体 CI/CD 平台实现”。
- 容器镜像可以作为优先兼容的交付形式，但不能被写成唯一合法制品。
- 发布、安全、部署文档已经存在，因此新文档必须做职责分层，而不是吞并全部内容。
- 当前仓库应继续保持“目标态方向”和“已落地工程资产”之间的明确区分。

### 3.3 Success criteria
pre_design 应使后续 OpenSpec 文档能够稳定表达：
- CI/CD 的阶段模型和边界清晰。
- 默认门禁定义清晰，且与现有架构/测试/安全治理方向一致。
- 可发布制品的抽象足够稳定，不依赖特定平台。
- 受控部署与发布后验证、回滚入口形成最小闭环。
- CI/CD 主文档与发布/安全/部署文档之间职责清晰，不重复吞并。

## 4. Constraints / Invariants

### 4.1 Constraints
本次 pre_design 必须遵守以下约束：

1. **平台中立**
   - 只能定义能力模型、阶段边界和治理要求。
   - 不能预设具体 CI/CD 平台，也不能写平台专属实现细节。

2. **范围到受控部署为止**
   - 可以覆盖部署触发、环境准入、发布后验证、回滚入口。
   - 不进入多环境晋升编排与复杂发布策略。

3. **默认 CI 门禁按严格治理处理**
   - 架构/边界检查必须作为默认阻断项处理，不能仅作为提示信息。
   - 文档一致性与依赖/安全最低门禁应被视为交付准入的一部分，而非可选增强项。

4. **以规范/基线文档为主**
   - 本次重点是文档与治理约束沉淀。
   - 不要求同时产出执行手册、平台接入指南或值班操作说明。

5. **采用“主文档 + 专题基线承接”组织方式**
   - 新增一篇 CI/CD 主文档。
   - 保留现有发布、安全、部署文档的专题职责。
   - 避免职责重复与内容冲突。

6. **交付对象采用“可发布制品”抽象**
   - 可兼容容器镜像作为优先适配形式。
   - 但不能将镜像定义为唯一合法交付物。

### 4.2 Invariants
以下内容属于后续文档不得推翻的不变量：
- CI/CD 在本次 change 中被定义为“治理与交付闭环”，不是平台配置清单。
- 架构/边界检查是默认门禁，不是可有可无的增强项。
- 发布对象必须可追踪、可验证，并能与发布记录和回滚入口关联。
- 受控部署必须包含最小发布后验证要求。
- CI/CD 主文档不能吞并现有发布、安全、部署文档的专题职责。
- 未确认已落地的工程资产或平台能力不得被表述为既成事实。

## 5. Architecture direction

### 5.1 Recommended direction
推荐采用“CI/CD 主文档作为中枢，现有专题文档按职责承接”的结构。

该结构中：
- 新增一篇 CI/CD 主文档，负责定义：
  - CI/CD 阶段模型
  - 默认 CI 门禁
  - 可发布制品
  - 受控部署边界
  - 最小发布后验证
  - 回滚入口
  - 与现有发布/安全/部署文档的关系
- 保留现有专题文档各自职责：
  - `docs/release_change_and_rollback.md` 继续负责发布控制、验证与回滚原则。
  - `docs/security_baseline.md` 继续负责安全最低要求与相关原则。
  - `docs/deployment_environment_baseline.md` 继续负责环境分层与部署边界。
- 后续 OpenSpec 文档应围绕“新增一个治理中枢，而不是重写整个文档体系”展开。

### 5.2 Why this direction
这种结构可以：
- 复用仓库现有文档体系，降低重复定义。
- 让 CI/CD 位于跨文档的流程治理层，而不是成为某个已有专题文档的附录。
- 在不绑定平台的前提下，为未来具体流水线实现预留稳定映射空间。

## 6. Key decisions / Trade-offs

### 6.1 Confirmed decisions
1. 采用平台中立设计。
2. 范围到受控部署为止。
3. 新增 CI/CD 主文档，而不是只做分散补丁式更新。
4. 默认质量门禁按严格治理处理。
5. 架构/边界检查进入默认阻断门禁。
6. 交付对象使用“可发布制品”抽象。
7. 本次以规范与基线文档为主，不做操作手册。

### 6.2 Trade-offs considered
#### Option A: 主文档中枢型（推荐）
选择原因：
- 最符合“平台中立 + 严格治理 + 受控部署边界”的目标。
- 既新增统一入口，又不破坏现有发布/安全/部署文档职责。
- 能以较稳定的范围承载当前 change。

#### Option B: 分散补丁型
放弃原因：
- 容易导致职责重叠与口径不一致。
- 难以形成清晰的 CI/CD 统一入口。
- 后续 OpenSpec 和评审时不易定位哪些要求属于 CI/CD 主闭环。

#### Option C: 偏实施映射型
放弃原因：
- 更容易把 GitHub Actions、Jenkins 等平台实现提前卷入本次范围。
- 会削弱平台中立边界。
- 与当前“先做治理基线，再谈平台映射”的优先级不一致。

#### Why not make image the only artifact
- 会过早绑定容器平台前提。
- 不利于保持平台中立。
- 与当前仓库强调“区分容器化方向与已落地工程资产”的原则不一致。

## 7. Critical flows

### 7.1 CI flow
后续设计应表达一个平台中立的 CI 流程抽象：
1. 触发与代码变更相关的校验。
2. 执行格式/静态检查。
3. 执行测试与必要的构建校验。
4. 执行文档一致性、架构/边界、依赖/安全最低门禁。
5. 形成通过/阻断结论，为可发布制品生成与部署准入提供输入。

### 7.2 Artifact flow
后续设计应将“可发布制品”视为交付闭环的中心对象：
1. 通过默认门禁后，产出可发布制品。
2. 制品需具备可追踪、可验证、可关联发布与回滚的属性。
3. 若采用容器镜像，应将其视为一种优先兼容形式，而非唯一形式。

### 7.3 Controlled deployment flow
后续设计应表达最小受控部署闭环：
1. 满足部署触发条件。
2. 满足目标环境准入条件。
3. 进入部署。
4. 执行最小发布后验证。
5. 验证失败时具备明确回滚入口与责任边界。

## 8. OpenSpec mapping

### 8.1 proposal.md should cover
- 为什么现在需要引入平台中立的 CI/CD 主文档。
- 当前缺口与风险。
- 本次 change 拟新增的治理能力与文档产物。
- 对既有文档边界的影响。

proposal 不应擅自扩写为：
- 具体平台选型结论。
- 多环境晋升方案。
- 云资源与运维组织流程。

### 8.2 design.md should cover
- CI/CD 阶段模型。
- 默认质量门禁分层。
- 可发布制品模型。
- 受控部署边界。
- 最小发布后验证与回滚入口。
- CI/CD 主文档与已有文档的职责映射。
- 如果未来进入实现阶段，能力如何映射到具体平台。

design 不应擅自扩写为：
- 具体 YAML / Jenkinsfile / pipeline DSL。
- 平台专属目录结构。
- 未确认存在的镜像仓库、签名、扫描、审批系统。

### 8.3 tasks.md should cover
- 新增 CI/CD 主文档。
- 定向更新发布/安全/部署相关文档。
- 如后续进入实现阶段，再拆具体校验、测试、构建、文档一致性、架构检查等任务。

tasks 不应提前拆解为：
- 复杂平台接入任务。
- 多环境部署编排任务。
- 云侧基础设施搭建任务。

## 9. Generation guardrails

### 9.1 Must follow
后续模型必须遵守：
- 平台中立。
- 范围只到受控部署。
- 默认质量门禁包含架构/边界检查，且按阻断门禁处理。
- 交付对象使用“可发布制品”抽象。
- 新增主文档并联动现有发布/安全/部署文档。
- 不把未落地资产写成已存在能力。

### 9.2 Allowed to elaborate
允许补全：
- CI/CD 阶段命名与分层细化。
- 门禁项的合理归类。
- 主文档与现有文档的章节级职责拆分。
- 与现有架构/测试/安全文档的交叉引用方式。

### 9.3 Forbidden to invent
禁止补全：
- 擅自绑定 GitHub Actions / GitLab CI / Jenkins。
- 擅自引入多环境晋升策略。
- 把镜像作为唯一合法交付物。
- 把企业级扫描、签名、审批链写成默认已具备。
- 把本 change 扩张成完整运维平台设计。

## 10. Scope fitness
当前判断该主题适合单个 change 承载。原因是：
- 当前目标清晰聚焦于 CI/CD 治理基线与相关文档联动。
- 已明确排除平台实现、多环境晋升与运维流程。
- 通过“主文档 + 专题基线承接”的方式可以控制范围，不需要拆分为多个 change。

## MODIFIED Requirements

### Requirement: 项目必须定义平台中立的 CI/CD 阶段模型
系统 MUST 提供一个平台中立的 CI/CD 基线定义，用于表达代码变更从校验、构建、可发布制品确认到受控部署的阶段边界与职责分工。该定义 MUST 描述哪些环节属于 CI，哪些环节属于受控部署范围内的 CD，并且 MUST 通过一篇 CI/CD 主文档统一承载阶段模型、默认门禁、可发布制品、部署边界与文档职责映射。该定义 MUST NOT 绑定 GitHub Actions、GitLab CI、Jenkins 或其他具体平台实现。

#### Scenario: 仓库以平台中立方式表达 CI/CD 阶段边界
- **WHEN** 团队查阅 CI/CD 基线文档或相关 OpenSpec 设计
- **THEN** 能明确区分校验、构建、可发布制品生成或确认、部署触发、部署验证与回滚入口等阶段职责
- **AND** 这些描述不依赖任何特定平台语法或平台专属功能名称

#### Scenario: CI/CD 主文档作为阶段模型中枢
- **WHEN** 团队补充或重构 CI/CD 相关文档
- **THEN** 阶段模型、默认门禁、可发布制品与部署边界由一篇主文档统一表达
- **AND** 发布、安全、部署专题文档继续承接各自专题细节，而不是被主文档吞并

#### Scenario: 后续实现可映射到不同流水线平台
- **WHEN** 团队后续需要把该基线映射到具体 CI/CD 平台
- **THEN** 可以在不改变阶段边界与治理要求的前提下完成平台适配

### Requirement: CI 默认门禁必须覆盖工程治理基线
系统 MUST 将默认 CI 门禁定义为交付准入条件的一部分，至少覆盖格式/静态检查、测试、构建校验、文档一致性校验、架构/边界检查，以及依赖/安全最低门禁。架构/边界检查 MUST 作为默认阻断门禁，文档一致性与依赖/安全最低门禁也 MUST 被视为默认交付准入的一部分，而不是可选增强项。

#### Scenario: 变更在进入可发布阶段前通过默认门禁
- **WHEN** 一次代码或文档变更进入默认 CI 流程
- **THEN** 该变更必须经过格式或静态检查、测试与必要的构建校验
- **AND** 必须经过文档一致性、架构/边界与依赖/安全最低门禁检查

#### Scenario: 架构边界检查被视为默认阻断条件
- **WHEN** 变更破坏了既定的架构边界或依赖方向约束
- **THEN** 默认 CI 门禁应将其视为阻断项，而不是仅作为提示信息

#### Scenario: 文档与安全最低门禁属于默认准入条件
- **WHEN** 变更未满足文档一致性要求或依赖/安全最低门禁
- **THEN** 默认 CI 门禁应阻止其进入可发布阶段
- **AND** 不应将这些检查降级为仅供参考的可选项

### Requirement: 交付闭环必须围绕可发布制品建立
系统 MUST 将“可发布制品”定义为交付闭环中的核心对象。可发布制品 MUST 可追踪、可验证，并可与发布和回滚关联。系统 MAY 兼容容器镜像作为优先适配形式，但 MUST NOT 将镜像定义为唯一合法交付物，也 MUST NOT 假定镜像仓库、签名、扫描或审批链已经落地。

#### Scenario: 通过门禁后产出可发布制品
- **WHEN** 一次变更通过默认 CI 门禁
- **THEN** 流程应进入可发布制品的生成或确认阶段
- **AND** 该制品可以与后续发布记录、验证结果和回滚入口建立关联

#### Scenario: 容器镜像不是唯一交付物前提
- **WHEN** 团队阅读 CI/CD 基线或生成后续设计文档
- **THEN** 可以看到容器镜像是一种兼容的交付形式
- **AND** 不会将其误解为唯一合法的制品形态或当前仓库已具备的既成能力

### Requirement: CD 范围必须止于受控部署闭环
系统 MUST 将本 change 的 CD 范围限定在“受控部署”为止，至少覆盖部署触发、目标环境准入、发布后最小验证与回滚入口。系统 MUST NOT 在该 change 中扩展到多环境晋升编排、复杂灰度策略、云资源拓扑或运维 SOP。

#### Scenario: 受控部署包含最小闭环
- **WHEN** 团队依据该基线定义后续部署流程
- **THEN** 流程中包含部署触发条件、目标环境准入条件、发布后最小验证以及回滚入口

#### Scenario: 多环境晋升不属于本次变更范围
- **WHEN** 有人尝试在该 change 中加入 test/dev/prod 晋升链路、审批流或复杂发布编排
- **THEN** 规格能够明确判定这些内容超出本次受控部署边界

### Requirement: CI/CD 主文档必须与现有基线文档分层协作
系统 MUST 通过新增或重构一篇 CI/CD 主文档来承载阶段模型、默认门禁、可发布制品、受控部署边界与文档职责映射；同时 MUST 保持 `docs/release_change_and_rollback.md`、`docs/security_baseline.md`、`docs/deployment_environment_baseline.md` 各自专题职责不被吞并。后续文档补充 MUST 采用定向联动方式，而不是由 CI/CD 主文档重写全部发布、安全或部署内容。

#### Scenario: CI/CD 主文档作为治理中枢
- **WHEN** 团队查阅新增或重构后的 CI/CD 相关文档
- **THEN** 可以从主文档获得统一的 CI/CD 阶段与交付闭环定义
- **AND** 可以看到它与发布、安全、部署文档之间清晰的职责关系

#### Scenario: 现有专题文档职责保持稳定
- **WHEN** 团队补充发布、安全或部署文档
- **THEN** 这些文档仍各自负责专题基线
- **AND** 不会因为 CI/CD 主文档的引入而失去原有职责边界

### Requirement: 基线文档不得将未落地能力描述为现成事实
系统 MUST 明确区分“当前仓库已存在的工程资产”与“后续可映射的目标能力”。后续 OpenSpec 文档和配套基线文档 MUST NOT 将具体 CI/CD 平台、企业级扫描/签名/审批链、多环境晋升能力、云平台交付资产或其他未确认已落地的能力写成现成事实。

#### Scenario: 文档保持对当前仓库事实的忠实表达
- **WHEN** 团队根据该 change 生成 proposal、design、tasks 或文档补充
- **THEN** 文本只描述当前已知存在的仓库资产与被确认的治理目标
- **AND** 不会把未来能力、平台设施或组织流程写成现状

#### Scenario: 未落地平台能力不会成为默认前提
- **WHEN** 团队参考该变更设计后续实现或评审新增文档
- **THEN** 不会将镜像扫描、制品签名、审批系统、平台集成或平台专属触发器视为默认存在

## Context

当前仓库已存在 `docs/cicd_baseline.md`、`docs/release_change_and_rollback.md`、`docs/security_baseline.md`、`docs/deployment_environment_baseline.md` 等治理文档，但这些内容目前仍以专题分散承载，缺少一个以“CI → 可发布制品 → 受控部署”闭环为主线的统一中枢。与此同时，仓库现状并不能支持把 GitHub Actions、GitLab CI、Jenkins、镜像仓库、签名、扫描、审批链等平台能力表述为已落地事实，因此本次设计必须保持平台中立，并以治理基线而不是平台实现为核心。

本次 change 的设计目标，是在不扩大到多环境晋升、复杂发布策略、云资源或运维操作体系的前提下，为后续文档补充与未来实现映射提供一组稳定约束：哪些阶段属于默认 CI/CD 闭环、哪些门禁必须默认阻断、什么是可发布制品、何时允许进入部署，以及部署后最低限度必须验证什么。

## Goals / Non-Goals

**Goals:**
- 为 `cicd-baseline` capability 细化一个平台中立的 CI/CD 治理模型。
- 定义 CI/CD 主文档应承载的范围：阶段模型、默认质量门禁、可发布制品、受控部署边界、最小发布后验证、回滚入口、文档职责映射。
- 明确 `docs/cicd_baseline.md` 与发布、安全、部署相关文档之间的职责分层与引用关系。
- 将架构/边界检查、文档一致性、依赖/安全最低门禁明确纳入默认阻断门禁。
- 为后续 tasks 拆解提供稳定文档治理范围，避免漂移到平台实现或运维体系设计。

**Non-Goals:**
- 不绑定任何具体 CI/CD 平台或平台专属语法。
- 不生成 GitHub Actions YAML、Jenkinsfile、GitLab CI 配置等实现产物。
- 不展开多环境晋升、复杂灰度、蓝绿、金丝雀或审批链设计。
- 不把镜像仓库、签名、扫描、审批系统等企业平台能力写成默认已具备。
- 不扩张为完整运维平台、运行手册或云资源拓扑设计。

## Decisions

### 1. 采用“CI/CD 主文档 + 专题基线承接”结构

**决策**：由 `docs/cicd_baseline.md` 承担 CI/CD 闭环治理中枢职责，负责定义阶段模型、默认门禁、可发布制品、受控部署边界、最小发布后验证与回滚入口；发布、安全、部署专题文档继续保留各自边界。

**原因**：这样既能提供统一入口，又不会吞并现有文档职责，符合当前仓库已具备专题文档、但缺少统一治理中枢的现实。

**备选方案**：
- 仅在各专题文档里做分散补丁：会继续放大职责重叠与口径不一致问题。
- 让 CI/CD 文档重写发布/安全/部署内容：会破坏现有专题职责并增加冲突风险。

### 2. 将默认质量门禁定义为阻断式交付准入

**决策**：将格式/静态检查、测试、必要构建校验、文档一致性、架构/边界检查、依赖/安全最低门禁统一视为默认交付准入条件，其中架构/边界检查必须按阻断项处理。

**原因**：`pre_design` 已明确严格治理是本次 change 的核心方向，若把这些项降为提示信息，后续实现和评审将重新回到口径分散状态。

**备选方案**：
- 将文档一致性、架构检查、安全最低门禁作为可选增强：不符合严格治理约束。
- 仅定义测试与构建为默认门禁：无法表达当前仓库强调的治理性要求。

### 3. 使用“可发布制品”作为交付对象抽象

**决策**：在设计中继续以“可发布制品”作为交付中心对象，要求其可追踪、可验证、可关联发布与回滚；容器镜像仅作为优先兼容形式，而不是唯一形式。

**原因**：该抽象既能兼容当前仓库对容器化方向的表达，又能避免把镜像仓库或容器平台前提写成既有事实。

**备选方案**：
- 直接把镜像定义为唯一交付物：会破坏平台中立和事实忠实性。
- 不定义统一交付对象：会让发布验证与回滚入口失去闭环核心。

### 4. CD 范围限定为受控部署最小闭环

**决策**：设计只覆盖部署触发条件、目标环境准入、发布后最小验证与回滚入口，不延伸到多环境晋升、复杂发布编排或云侧部署架构。

**原因**：这能满足“到受控部署为止”的范围约束，同时提供足以支撑后续实现映射的最小闭环。

**备选方案**：
- 扩展到 dev/test/prod 晋升、审批流与灰度策略：超出本次范围。
- 只定义 CI 不定义部署：会使闭环断在可发布制品之前，不满足目标。

### 5. 以后续实现映射作为说明性内容，而不是当前事实

**决策**：可以在设计中说明这些治理能力未来如何映射到具体平台，但不能把具体平台目录、DSL、平台能力或企业基础设施写成当前仓库已具备的事实。

**原因**：这同时满足平台中立、事实忠实性和未来可映射性三项要求。

**备选方案**：
- 完全不讨论未来映射：会削弱设计对后续落地的指导价值。
- 提前给出平台级目录结构或实现草案：会把平台实现错误带入当前范围。

## Risks / Trade-offs

- [CI/CD 主文档与专题文档边界仍可能模糊] → 通过在设计与 tasks 中显式定义每篇文档的职责与引用关系来降低重复风险。
- [默认门禁定义过于原则化，后续实现口径仍可能分散] → 在 spec 中把门禁分类、阻断属性和文档职责映射写成规范性要求。
- [可发布制品抽象过宽，后续实施时解释不一致] → 要求其至少满足可追踪、可验证、可关联发布与回滚三项稳定属性。
- [设计被误读为平台实现蓝图] → 在文档中持续强调平台中立与“未确认能力不得写成既成事实”的边界。
- [受控部署范围被后续扩张到多环境晋升或运维体系] → 在 spec 与 tasks 中把这些内容明确列为非目标和禁止扩展项。

## Migration Plan

1. 在本 change 中先为 `cicd-baseline` 产出 delta spec，固化 CI/CD 主文档职责、默认门禁、可发布制品与受控部署边界。
2. 基于 spec 生成 tasks，聚焦文档治理动作：新增或重构 CI/CD 主文档内容，并对发布、安全、部署文档进行定向联动。
3. 后续若进入实现阶段，再由新的 change 或后续任务将这些治理要求映射到具体流水线平台，而不是在本次 change 中直接落平台配置。
4. 回滚策略为：若发现新主文档与现有专题文档职责冲突，应回退到“CI/CD 主文档仅保留治理中枢定义，专题细节继续保留在原文档”的更小边界，而不是扩大主文档范围。

## Open Questions

- `docs/cicd_baseline.md` 是以增量重构方式加强，还是保留现有内容并按章节重组，需要在 tasks 里进一步明确。
- 主文档与 `docs/release_change_and_rollback.md`、`docs/security_baseline.md`、`docs/deployment_environment_baseline.md` 的交叉引用粒度，采用章节级引用还是原则级引用，还需在文档拆解时统一。
- 是否需要在本次 change 中同时补一份面向评审者的“CI/CD 文档边界检查清单”，目前可以作为后续可选补充，但不应影响 apply-ready 状态。

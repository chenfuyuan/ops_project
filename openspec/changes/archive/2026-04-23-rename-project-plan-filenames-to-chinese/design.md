## Context

当前 `novel_docs/project_plan/` 已经作为小说项目规划阶段的唯一主入口，但目录内仍保留大量英文 slug 文件名，例如 phase 总览、requirements 索引页和各 requirement card。文档正文已经以中文为主，文件系统命名却仍是英文，这会增加人工浏览成本，也让后续基于中文语境继续维护规划资料时产生不一致。

这次变更只作用于文档资产，不涉及运行时代码、接口、依赖或数据结构。实现重点不是内容设计，而是以最小扰动方式完成批量重命名，并确保所有相对链接在重命名后仍然可用。

## Goals / Non-Goals

**Goals:**
- 将 `novel_docs/project_plan/` 下目标 Markdown 文件重命名为语义对应的中文文件名
- 同步修正所有受影响的相对链接，保证目录导航可继续工作
- 保持既有目录层级、phase 边界、requirement card 数量与文档语义不变
- 让后续规划维护统一采用中文阅读与命名语境

**Non-Goals:**
- 不重写 phase 或 requirement card 的正文内容
- 不调整目录结构，不新增或删除 phase、requirements、card
- 不把规划文档扩展成实现设计、任务清单或技术方案
- 不涉及 `novel_docs/project_plan/` 之外的实现代码或运行时行为

## Decisions

### 决策 1：只对文件名做语义等价的中文替换，不改变目录结构
- 选择：保留 `novel_docs/project_plan/` 的现有目录层级，仅将目标 `.md` 文件名从英文 slug 改为中文名。
- 原因：用户需求是“把这个目录下的文件名都换成中文”，最小实现路径就是保留结构、只改命名，这样最不容易引入额外范围。
- 备选方案：连同目录名一并中文化。
- 不采用原因：目录名变更会显著扩大影响面，增加引用修复和潜在外部依赖风险，也超出当前需求表述。

### 决策 2：在同一次变更中同步修复全部内部相对链接
- 选择：凡是 `project_plan` 内部文档引用到被重命名文件的地方，都在本次一起更新。
- 原因：如果只改文件名不改链接，文档导航会立即失效，无法满足“重命名后仍可使用”的基本要求。
- 备选方案：先只重命名文件，后续再补链。
- 不采用原因：会制造中间失效状态，不适合这种纯文档一致性变更。

### 决策 3：将 requirement card 的中文文件名作为稳定展示名
- 选择：每张 requirement card 使用与其中文业务名称一致或高度对应的中文文件名。
- 原因：这些卡片本身就是给中文场景下的人读和挑选的，文件名应直接表达业务含义，减少英文 slug 到中文标题的额外心智映射。
- 备选方案：保留英文文件名，仅调整标题。
- 不采用原因：无法满足本次明确的文件名中文化目标。

## Risks / Trade-offs

- [风险] 批量重命名后遗漏某些相对链接，导致文档导航断裂 → [缓解] 先建立旧文件名到新文件名的完整映射，再统一搜索并逐项替换引用
- [风险] 中文文件名在少数外部工具或脚本中兼容性较弱 → [缓解] 将变更范围限制在规划文档目录内，并在落地前确认当前仓库工作流主要围绕人工阅读与版本管理，而非依赖固定英文路径脚本
- [风险] 某些英文 slug 存在多种中文翻译方式，导致命名不一致 → [缓解] 以当前文档标题和业务术语为准，优先选用正文已出现的中文名称并在同类文件中保持一致
- [权衡] 中文文件名提升可读性，但会降低与历史英文 slug 的直接路径兼容 → [缓解] 接受一次性路径切换成本，通过同步修复链接和后续统一命名来换取长期可读性

## Migration Plan

1. 盘点 `novel_docs/project_plan/` 下所有待重命名文件，形成英文到中文的映射表。
2. 逐个执行文件重命名，保持原目录层级不变。
3. 搜索并更新 `project_plan` 范围内所有受影响的相对链接与引用。
4. 逐页检查总览页、phase 索引页和 requirement card 导航是否仍然可读、可定位。
5. 如发现不合适的中文命名，可在本地继续调整映射后再整体提交。

回滚方式：基于映射表将中文文件名恢复为原英文文件名，并回退对应链接修改。

## Open Questions

- 少数 requirement card 若标题较长，文件名应采用完整中文标题还是更短但不失真的中文概括，需要在落地时统一尺度。

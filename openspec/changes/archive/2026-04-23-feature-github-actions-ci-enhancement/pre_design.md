# pre_design：增强 GitHub Actions CI 的质量门禁与架构守卫

## 1. Problem framing

### 要解决的问题
当前仓库已经在 `/.github/workflows/ci.yml` 中具备最小可用的 GitHub Actions CI，能够覆盖静态检查、架构检查、单元测试与集成测试这三段基础链路。

但现状仍停留在“基础校验流水线”层面，还不足以承担更稳定的工程化守门职责，主要缺口包括：
- 缺少面向 Pull Request 协作场景的明确质量门禁。
- 缺少将“架构守卫”作为一等 CI 目标的显式设计表达。
- 缺少缓存、并发治理、轻量路径感知等效率增强手段。
- 缺少测试结果、失败上下文与诊断产物的可见性设计。
- 缺少对“覆盖强度”与“反馈速度”之间平衡的明确取舍。

### 背景与触发原因
仓库内已有以下事实基础：
- `/.github/workflows/ci.yml` 已落地最小 CI，采用静态/结构检查、测试执行、运行时验证的三段顺序。
- `ai_docs/rules/testing_rules.md` 已明确架构测试应承担边界守护职责，而不是制造维护噪音。
- `ai_docs/rules/architecture_rules.md` 已明确 `service -> ports -> infrastructure` 等关键依赖边界。
- `pyproject.toml` 已明确 Python 3.13 与 `uv` 为依赖管理和命令执行基线。

因此，本次 change 的核心不是“从零创建 CI”，而是在已有最小 CI 的基础上，增强其质量门禁、架构守卫、运行效率与失败可诊断性。

### 当前方案/现状的核心不足
当前 `ci.yml` 的核心不足不在于没有基础校验，而在于：
- 质量门禁能力不足，PR 协作约束还不够前置自动化。
- 架构守卫虽已存在测试入口，但尚未被提升为清晰的 CI 治理对象。
- 缺少缓存与并发控制，重复执行成本偏高。
- 缺少报告与失败归档设计，导致问题定位效率有限。
- 缺少轻量路径感知，可能让部分无关变更承担不必要执行成本。

### 为什么现在做
仓库已经进入“有最小 CI、开始追求稳定协作质量”的阶段。如果继续停留在最小流水线层，CI 只能发现一部分问题，却难以成为稳定、低噪音、可解释的准入门禁。现在补齐增强层，能够在不引入过重治理负担的前提下，让现有 CI 更适合日常 PR 协作与后续渐进演进。

## 2. Goals / Non-goals

### Goals
本次 change 要达成以下目标：
- 将当前最小 GitHub Actions CI 提升为平衡型工程化 CI。
- 在不进入发布或部署链路的前提下，补齐高收益、低复杂度的质量门禁。
- 将架构守卫明确纳入 CI 的一等职责，而不只是顺带执行一组测试。
- 提升 Pull Request 阶段的反馈质量、失败可诊断性与执行效率。
- 为后续 proposal、design 与 tasks 提供稳定护栏：本次重点是 **CI 校验 + PR 质量门禁 + 架构守卫**。

### Non-goals
本次 change 明确不做以下事项：
- 不做发布、部署、镜像构建、制品上传或环境验证。
- 不做完整 CD 流程设计。
- 不引入与当前仓库成熟度不匹配的重型治理能力。
- 不将多版本、多平台、多维矩阵测试作为本次默认目标。
- 不围绕本次主题之外的问题进行代码结构重构。

### 看起来相关但本次不纳入的内容
以下事项可能主题相关，但不属于本次范围：
- Docker 镜像构建、镜像扫描与发布前检查。
- 制品打包、上传、签名、供应链治理。
- 环境部署校验、发布审批、回滚入口。
- 复杂 CODEOWNERS 审批治理或团队级工作流编排。
- 与本次 CI 增强无直接关系的仓库重构。

## 3. Requirement understanding

### 对需求的深层理解
本次需求不是简单“再给 CI 多加几个 job”，而是要把当前最小流水线提升到“能稳定守门”的层级。其本质是：
- 让 CI 在 PR / push 阶段更早、更清晰地暴露质量问题。
- 让质量门禁把原本依赖人工 review 的一部分基础要求前置自动化。
- 让架构守卫持续保护仓库中已经明确定义的高价值边界。
- 让增强后的 CI 仍然保持反馈速度与维护成本的可接受平衡。

### 隐含约束
结合现有仓库与用户已确认方向，可推导出以下隐含约束：
- 本次增强必须以当前 `ci.yml` 为演进基础，而不是推翻重做。
- 新增门禁必须是可解释、低噪音的，不能让开发者难以判断失败原因。
- 架构守卫应聚焦高价值、稳定的边界，而不是把所有规范都做成硬性门禁。
- 效率增强与质量增强要同时考虑，不能只追求覆盖而忽视反馈速度。
- 方案要符合“平衡优先”原则，避免把仓库过早推进到重型流水线治理阶段。

### 成功标准
本次 change 的成功标准是：
- GitHub Actions CI 在现有三段基础上，新增明确的 PR 质量门禁与架构守卫表达。
- CI 失败时能更快定位问题，具备基础报告或失败上下文归档能力。
- 执行效率较当前方案得到改善或至少被合理控制，例如通过缓存、并发取消或轻量路径感知抵消新增开销。
- 新增门禁总体保持高收益、低复杂度，符合“平衡优先”。
- 不扩大到发布、部署或与本次范围无直接关系的治理问题。

### 对业务/系统边界的解释
本次 change 处于工程治理与协作准入层，不是业务功能或系统运行时架构调整。它应服务于代码准入、结构守护和协作质量，不应改变业务分层本身，也不应侵入发布交付边界。

## 4. Constraints / Invariants

### 必须遵守的约束
- 本次仅覆盖 GitHub Actions 下的 CI 校验链路与 PR 质量门禁。
- 继续以 `uv` 作为依赖管理与命令执行基线。
- 继续以 Python 3.13 为默认运行基线，不默认引入重型矩阵。
- 增强后不能显著拉长日常 PR 反馈时间。
- 新增规则必须稳定、可解释、低噪音，避免频繁误报。

### 关键不变量
后续 OpenSpec 文档与实现阶段不得推翻以下不变量：
- 现有三段核心职责必须保留：静态/结构检查、测试执行、运行时验证。
- 架构守卫必须被视为显式的 CI 目标，而不是边缘性的附带检查。
- PR 质量门禁必须以前置、低噪音、可解释为原则。
- 增强项必须优先选择高收益、低复杂度方案。
- 后续文档不得将发布、部署、镜像、制品或完整 CD 混入本 change。

### 容易被误解的硬边界
- 这是“增强现有 GitHub Actions CI”，不是“重新定义 CI 基线”。
- 这是“平衡型工程化增强”，不是“追求覆盖极致的重型治理”。
- 这是“CI 校验 + PR 质量门禁 + 架构守卫”，不是“完整 PR 平台治理”或“CI/CD 一体化建设”。
- 允许做轻量路径感知，但不以复杂影响分析为目标。

## 5. Architecture direction

### 推荐的总体架构方向
采用 **单一主 CI workflow 下的分层守门模型**，继续以 `/.github/workflows/ci.yml` 作为主入口，而不是拆成多个分散 workflow。

建议将增强后的检查职责稳定分为四个层次：
1. **质量预检层**：PR 标题、基础规范、格式检查。
2. **静态与架构守卫层**：lint、结构边界检查、架构测试。
3. **测试验证层**：单元测试、集成测试及必要的测试结果汇总。
4. **诊断与运行治理层**：缓存、并发取消、失败产物归档、必要的轻量路径感知。

### 关键模块边界
建议后续设计围绕以下稳定职责边界展开：
- 主 workflow 仍为 `.github/workflows/ci.yml`。
- 质量预检层负责轻量、前置、低成本的协作准入规则。
- 静态与架构守卫层负责表达并保护关键结构边界。
- 测试验证层负责维持现有单元测试、集成测试路径的门禁语义。
- 运行治理层负责降低重复执行成本并提高失败可诊断性。

### 职责划分原则
- workflow 负责组织执行顺序、门禁语义与治理能力。
- 架构守卫负责表达高价值结构边界，而不是泛化为所有代码规范。
- 质量门禁负责前置常规协作要求，而不是替代人工评审。
- 运行治理负责提高效率与反馈质量，而不是扩展到发布流程。

### 哪些边界必须稳定
以下边界应在后续设计与实现中保持稳定：
- 单 workflow 主入口，不默认拆成多个 workflow。
- 现有三段核心校验职责不消失，只允许在其上增强。
- 架构守卫必须作为独立设计对象被清晰表达。
- 不把发布、部署或重型治理能力并入本次 change。

## 6. Key decisions / Trade-offs

### 已确认的关键决策
1. **采用渐进增强，而不是推倒重做**
   - 在现有 CI 基础上增强，比重构整套流水线更稳妥。

2. **采用方案 B：渐进增强型 CI**
   - 在现有基础上补齐缓存、格式检查、PR 质量门禁、测试可见性、失败产物归档、并发治理与轻量路径感知。

3. **将架构守卫列为一等公民**
   - 不仅保留现有架构测试入口，还要在后续设计中明确其属于 CI 的必过门禁之一。

4. **坚持“平衡优先”**
   - 优先补高收益、低复杂度增强项，不追求一次到位的极致覆盖。

5. **不引入超出当前成熟度的重型治理**
   - 暂不纳入多 Python 版本矩阵、多平台矩阵、复杂审批自动化、发布前 build 验证等能力。

### 为什么这样选
- 与仓库现状契合，能直接复用已存在的最小 CI 资产。
- 能同时改善质量门禁、结构守护、失败诊断和执行效率。
- 在不明显增加协作负担的前提下，提升 PR 准入质量。
- 符合“平衡优先”的目标，避免 CI 很快膨胀为难以维护的治理系统。

### 明确放弃的替代方向
本次 change 明确不采用以下方向：
- **最小增强方案**：收益不足，无法充分补齐质量门禁与诊断能力。
- **严格质量门禁型方案**：治理强度过高，容易超出当前仓库成熟度与本次边界。
- **多 workflow 分拆方案**：会增加理解、排障与维护复杂度。
- **重型矩阵或发布前校验方案**：与当前范围不匹配，且会显著增加 CI 成本。

## 7. Critical flows

### 核心控制流
增强后的核心控制流应表达为：
1. 代码变更触发主 CI workflow。
2. 先进入质量预检层，快速失败不符合基础准入要求的变更。
3. 进入静态与架构守卫层，阻断边界违规与静态质量问题。
4. 进入测试验证层，验证单元测试与集成测试链路。
5. 在运行治理与诊断层补充缓存、并发控制与失败归档能力。
6. 任一强门禁失败，则整体 CI 失败。

### 关键数据流 / 配置流
- `pyproject.toml` 提供 Python 依赖与 `uv` 执行基线。
- `/.github/workflows/ci.yml` 负责承接触发条件、执行顺序与治理增强。
- `tests/architecture`、`tests/unit`、`tests/integration` 继续作为主要自动化校验入口。
- PR 元数据（如标题）成为质量预检层的输入之一。
- 测试报告与失败日志成为诊断层的主要输出之一。

### 重要异常路径
需要在后续 design 中显式考虑的异常路径包括：
- 缓存策略不当导致收益不明显或引入稳定性问题。
- 轻量路径感知配置不当导致应运行的检查被跳过。
- 架构守卫规则过细，造成误报或维护噪音。
- 新增 PR 门禁规则不清晰，导致开发者难以理解失败原因。
- 失败归档不足以支撑问题定位，导致新增治理价值有限。

这些异常路径应在后续设计中被处理，但不应导致范围外扩。

## 8. OpenSpec mapping

### `proposal.md` 应承接什么
后续 proposal 应重点表达：
- 为什么当前最小 CI 需要增强为平衡型工程化 CI。
- 当前缺口不仅在于 workflow 能否运行，还在于质量门禁、架构守卫、效率治理与失败诊断不完整。
- 本次 change 的边界是 **CI 校验 + PR 质量门禁 + 架构守卫**。
- 本次不进入发布、部署、镜像、制品或完整 CD。

### `design.md` 应承接什么
后续 design 应重点展开：
- `/.github/workflows/ci.yml` 的增强型结构如何组织。
- 质量预检层、静态/架构守卫层、测试层、运行治理层如何协作。
- 哪些检查属于必过门禁，哪些属于辅助诊断。
- 架构守卫应固化哪些高价值规则。
- 如何通过缓存、并发取消、轻量路径感知来维持“平衡优先”。

### `tasks.md` 应承接什么
后续 tasks 应围绕以下主题拆解：
- workflow 增强任务。
- PR 质量门禁任务。
- 架构守卫补强任务。
- 报告、归档与反馈质量增强任务。
- 必要的说明文档或配置补充任务。

### 哪些内容不应由后续文档自行扩写
后续文档不得自行扩写以下内容：
- 发布、部署、回滚相关流程。
- 镜像构建、制品上传、供应链发布治理。
- 与当前成熟度不匹配的复杂多维矩阵或重型平台治理。
- 推翻“渐进增强 + 平衡优先 + 架构守卫显式化”的既定方向。

## 9. Generation guardrails

### 必须遵守
后续生成 proposal、design、tasks 时必须遵守：
- 本次 change 的核心主题是：**增强 GitHub Actions CI，使其具备平衡型质量门禁与架构守卫能力**。
- 必须保留现有三段核心校验职责，并在其上做增强。
- 必须把架构守卫视为显式门禁目标。
- 必须以 `uv`、Python 3.13 与当前仓库成熟度为基线。
- 必须优先选择高收益、低复杂度增强项。

### 允许补全
后续文档允许补全的内容包括：
- 具体 job 组织方式与依赖关系。
- PR 标题门禁、格式检查、报告归档等常规工程化细节。
- 轻量路径感知、缓存与并发治理的具体实现方式。
- 架构守卫的具体规则清单，只要不偏离高价值稳定边界。

### 禁止补全
后续文档禁止擅自补全以下内容：
- 擅自扩展到发布、部署、镜像构建、制品上传或完整 CD。
- 擅自引入与现阶段不匹配的重型治理，如复杂多维矩阵或过度审批自动化。
- 擅自把低价值、易误报规则包装成默认强门禁。
- 擅自把未确认事项写成既定事实。

### 遇到未决事项时的处理原则
如果后续生成 proposal、design 或 tasks 时遇到未决事项，应遵循以下原则：
- 可以补充不改变核心方向的常规实现细节。
- 不能通过脑补扩大范围或改变边界。
- 如果某事项会影响“平衡优先、渐进增强、架构守卫显式化”的核心判断，应显式标记为待决，而不是擅自定案。

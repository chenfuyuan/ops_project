## Context

当前仓库已经通过 `/.github/workflows/ci.yml` 落地了最小 GitHub Actions CI，能够覆盖静态/结构检查、测试执行与运行时验证三段基础职责。与此同时，`ai_docs/rules/architecture_rules.md` 与 `ai_docs/rules/testing_rules.md` 已明确架构守卫应承担高价值边界保护职责，`pyproject.toml` 也已固定 Python 3.13 与 `uv` 基线。

本次设计的背景不是“是否需要 CI”，而是“如何在不引入重型治理的前提下，让现有最小 CI 具备更稳定的 Pull Request 准入能力”。因此，设计必须同时满足四类目标：
- 提升 PR 协作时的质量前置能力。
- 将架构守卫显式化，而不是把它隐藏在测试执行细节中。
- 通过缓存、并发治理与轻量路径感知控制新增开销。
- 提供更好的失败诊断信息，降低排障成本。

本次变更只针对 GitHub Actions 下的 CI 与 PR 质量门禁，不涉及发布、部署、镜像或制品交付。

## Goals / Non-Goals

**Goals:**
- 在现有最小 GitHub Actions CI 基础上增加平衡型工程化增强能力。
- 引入轻量且可解释的 PR 质量门禁，如标题规范与格式检查。
- 将架构守卫提升为显式门禁层的一部分，并保持其聚焦高价值、稳定边界。
- 增加缓存、并发取消、轻量路径感知、测试结果可见性与失败产物归档等治理能力。
- 保持 `uv`、Python 3.13、主 workflow 入口与三段核心职责不变。

**Non-Goals:**
- 不设计发布、部署、回滚、镜像构建或制品上传流程。
- 不引入多 Python 版本、多平台、多维矩阵等重型流水线治理。
- 不将低价值、易误报的规则升级为默认强门禁。
- 不改变现有业务代码、运行时架构或应用分层。

## Decisions

### 决策 1：继续采用单一主 workflow 分层增强，而不是拆分多个 workflow
- **选择**：继续以 `/.github/workflows/ci.yml` 作为唯一主入口，在单 workflow 内增加质量预检、静态/架构守卫、测试验证与运行治理能力。
- **原因**：当前仓库规模下，单 workflow 更利于理解、维护与排障，也更符合 `pre_design` 中“渐进增强、非推倒重做”的约束。
- **备选方案**：拆成多个 workflow 分担 PR 门禁、测试与诊断任务。
- **为什么不选**：虽然视觉上更细，但会增加触发条件、依赖关系与协作心智成本，不符合平衡优先。

### 决策 2：将质量预检层限定为轻量、前置、低噪音门禁
- **选择**：在最前面增加 PR 标题校验与格式检查等轻量门禁，优先拦截不符合基础协作约束的变更。
- **原因**：这些检查执行快、反馈早、解释性强，适合作为 PR 协作第一道门。
- **备选方案**：增加复杂 commit message 检查、标签治理或审批自动化。
- **为什么不选**：这些规则超出本次范围，且对当前仓库成熟度来说过重，容易制造流程负担。

### 决策 3：将架构守卫从“测试细节”提升为“显式门禁职责”
- **选择**：在设计上明确静态与架构守卫层承担架构边界保护职责，要求其成为必过门禁之一。
- **原因**：仓库当前已有 `tests/architecture`，但若不显式提升其职责，后续实现容易把它视为可弱化的附带检查。
- **备选方案**：保持现状，只在测试命令中继续顺带执行架构测试。
- **为什么不选**：这会削弱架构守卫在增强型 CI 中的治理可见性，不利于后续稳定演进。

### 决策 4：通过运行治理能力抵消新增质量门禁的执行成本
- **选择**：引入 `uv` / Python 依赖缓存、同分支并发取消、轻量路径感知以及失败产物归档。
- **原因**：本次设计需要在“增加门禁”与“控制耗时”之间取得平衡，治理增强是实现该平衡的必要配套。
- **备选方案**：只增加门禁，不处理执行效率与诊断能力。
- **为什么不选**：会导致 CI 变慢但体验没有同步提升，违背“平衡优先”目标。

### 决策 5：spec 层新增增强 capability，并仅对 minimal capability 做边界澄清
- **选择**：新增 `github-actions-enhanced-ci` capability，并对 `github-actions-minimal-ci` 做有限修改，明确其作为增强型 CI 基础入口的保留边界。
- **原因**：本次 change 并不是重写 minimal baseline，而是在其之上叠加增强能力；spec 层需要反映这一层次关系。
- **备选方案**：直接在 `github-actions-minimal-ci` 中塞入所有增强要求。
- **为什么不选**：会污染最小能力边界，使 minimal spec 不再保持“最小、稳定、平台映射”的定位。

## Risks / Trade-offs

- **[风险] 新增门禁过多导致 PR 反馈时间明显增长** → **缓解**：优先选择低成本检查，并通过缓存、并发取消与轻量路径感知抵消开销。
- **[风险] 架构守卫规则过细导致误报** → **缓解**：只固化高价值、稳定的边界，避免对命名或易变结构做脆弱校验。
- **[风险] PR 质量门禁定义不清，开发者难以判断修复方式** → **缓解**：仅保留可解释、可直接修复的规则，并在设计中要求失败信息清晰。
- **[风险] 路径感知配置错误导致应运行的检查被跳过** → **缓解**：采用轻量且保守的路径感知，宁可多跑少量检查，也不遗漏关键门禁。
- **[风险] 失败归档做得过重或过轻** → **缓解**：只归档对测试排障和 CI 失败定位有直接价值的报告与日志，避免上传无关产物。

## Migration Plan

1. 以现有 `/.github/workflows/ci.yml` 为基础，定义增强后仍保持单 workflow 主入口。
2. 明确新增的质量预检、架构守卫、测试可见性与运行治理职责，并映射到 workflow/job 或 step 设计。
3. 增量调整相关 spec，使 enhanced capability 与 minimal capability 的边界关系清晰可追踪。
4. 在实现阶段逐步增强 workflow，并通过现有测试目录验证增强后门禁仍符合三段核心职责。
5. 若增强后出现明显误报或执行成本异常，可按增强项逐项回退；不需要回滚业务代码或发布资产，因为本次变更不触碰运行时与交付链路。

## Open Questions

- PR 标题门禁最终采用何种规则集合，是否需要与仓库已有提交/PR 规范进一步对齐。
- 轻量路径感知的粒度应细化到哪些目录，才能在不漏检的前提下提供稳定收益。
- 失败产物归档具体应覆盖测试报告、日志还是两者，需在实现阶段结合实际输出资产判断。
- 架构守卫是否只依赖现有 `tests/architecture`，还是需要补充更明确的结构检查入口。

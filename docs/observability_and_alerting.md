# 可观测性与告警

## 文档目标
本文档定义项目最小可观测能力与告警基线，确保系统具备发现问题、定位问题和支撑发布验证的基本能力。

## 适用范围
- API 进程
- Worker 进程
- PostgreSQL、Redis、对象存储相关调用
- 后台任务与关键业务链路

## 本文负责 / 不负责
### 本文负责
- 定义日志、指标、健康检查、告警分级基线
- 定义发布后观察和故障定位所需的最小闭环
- 定义应优先建立的观测对象

### 本文不负责
- 不绑定具体监控平台或日志平台
- 不展开具体值班流程
- 不替代发布与回滚流程文档

## 可观测性目标
最小目标包括：
- 能发现系统不可用、错误率升高、积压或依赖异常
- 能从请求或任务追踪到对应日志与指标
- 能在发布后验证关键链路是否健康

## 日志基线
### 结构化日志
- 日志应优先采用结构化输出
- 应包含时间、级别、服务名、环境等基础字段

### 关联字段
- API 请求应具备 request_id 或等价字段
- Worker 任务应具备 task_id 或等价字段
- 跨进程链路应尽量保留关联标识

### 错误日志要求
- 错误日志应能反映失败位置、异常类型和上下文摘要
- 敏感字段必须脱敏
- 不输出明文密钥或完整凭据

## 指标基线
建议至少具备以下观测对象：

| 对象 | 最低关注项 |
| --- | --- |
| API | 请求量、延迟、错误率 |
| Worker | 存活状态、任务吞吐、失败率 |
| Queue | 队列积压、重试量 |
| PostgreSQL | 可达性、连接异常 |
| Redis | 可达性、超时或错误 |
| 对象存储 | 上传下载失败率 |

## 健康检查基线
- API 应提供最小健康检查接口
- 健康检查至少覆盖进程存活和核心依赖可达性判断策略
- Worker 应具备可确认其仍在消费任务的存活信号

## 告警分级
### P1
- 系统不可用
- 核心链路连续失败
- 队列严重积压且影响核心业务

### P2
- 错误率明显升高
- 关键依赖异常但系统仍部分可用
- 发布后关键指标异常波动

### P3
- 非核心异常、容量预警、可延后处理的噪音问题

## 发布后观察要求
每次正式发布后应观察：
- API 健康检查
- 关键请求错误率与延迟
- Worker 任务失败率
- 队列是否异常积压
- 是否出现新类型错误日志

## 角色与责任
- 开发：保证新增关键链路具备日志和必要指标
- 评审者：检查是否遗漏关键链路观测能力
- 发布责任人：在发布后观察核心指标和错误日志

## 校验要点
- 是否区分了 API、Worker、Queue 和核心依赖的观测对象
- 是否要求日志具备关联字段
- 是否定义了最低健康检查要求
- 是否定义了发布后观察范围和告警分级

## 例外与变更机制
如引入新的关键链路、后台任务类型或外部依赖，应同步补充对应日志、指标和告警策略；如暂不具备完整观测能力，应记录缺口和补齐时间。

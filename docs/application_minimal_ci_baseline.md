# 应用代码最小 CI 实践基线

## 文档目标
本文档定义应用代码的最小 CI 实践基线，用于在平台中立前提下收敛 `app/` 与直接相关测试的默认覆盖范围、最小流水线分层与默认阻断门禁。

## 适用范围
- `app/` 下的应用代码
- 与应用代码直接相关的 `tests/unit/`、`tests/integration/`、`tests/architecture/`
- 面向应用代码准入的最小 CI 校验实践

## 本文负责 / 不负责
### 本文负责
- 定义应用代码最小 CI 的默认覆盖对象
- 定义三层最小流水线模型
- 定义默认阻断门禁的最小集合
- 定义与 `docs/cicd_baseline.md` 的职责分层

### 本文不负责
- 不重写平台中立的 CI/CD 主基线
- 不定义文档一致性门禁
- 不定义受控部署、发布后验证、回滚入口或其他 CD 内容
- 不定义 GitHub Actions、GitLab CI、Jenkins 或其他平台配置语法
- 不假定镜像扫描、签名、审批链或企业平台能力已经落地

## 默认覆盖对象
应用代码最小 CI 默认覆盖以下范围：
- `app/`
- `tests/unit/`
- `tests/integration/`
- `tests/architecture/`

该范围用于表达应用代码的默认准入条件，不扩展到 docs/spec 一致性治理，也不替代发布或部署治理。

## 三层最小流水线模型
### 1. 静态与结构校验层
用于尽早阻断不满足应用代码默认准入条件的变更。

至少包括：
- 格式或静态检查
- 架构/边界检查
- 必要的依赖/安全最低检查

其中，架构/边界检查必须作为默认阻断项处理，不能降级为提示信息。

### 2. 测试校验层
用于验证应用代码在当前最小实践范围内的默认正确性。

至少包括：
- 单元测试
- 已纳入默认准入的集成测试
- 架构测试

该层应遵循现有测试分层规则，不应以单一重型端到端测试替代分层测试。

### 3. 可运行性与构建确认层
用于确认应用代码在进入后续交付阶段前满足最小可运行前提。

至少包括：
- 必要的构建校验
- 导入校验、装配校验或等价运行性确认

该层只负责最小可运行前提，不扩展为完整发布制品生成、部署触发或其他 CD 语义。

## 默认阻断门禁
应用代码最小 CI 的默认阻断门禁至少包括：
- 格式或静态检查失败
- 架构/边界检查失败
- 默认测试集合失败
- 必要的依赖/安全最低检查失败
- 最小可运行性或构建确认失败

推荐的最小命令入口可按三层模型表达为：
- 静态与结构校验：`uv run ruff check app tests`
- 架构边界检查：`uv run python -m unittest discover -s tests/architecture`
- 测试校验层默认测试集合：`uv run python -m unittest discover -s tests/unit`
- 可运行性与构建确认层：`uv run python -m unittest discover -s tests/integration`

是否纳入默认门禁，应优先以“最小必需、可稳定执行、与现有规则一致”为判断原则。

## 当前最小校验资产
当前仓库已具备以下可直接纳入应用代码最小 CI 的基础资产：
- `ruff` 作为格式或静态检查工具，并已通过 `pyproject.toml` 的 dev 依赖声明纳入仓库工具链
- `tests/architecture/test_architecture_rules.py` 作为架构/边界检查资产
- `tests/unit/test_api_bootstrap.py`、`tests/unit/test_database_bootstrap.py` 等作为应用代码默认单元测试资产
- `tests/integration/test_runtime_validation.py` 作为最小可运行性与装配校验资产

这些资产共同构成当前仓库可稳定执行的最小应用代码 CI 基础集合，后续扩展应在不破坏三层模型与默认范围的前提下进行。

## 三层默认执行顺序
应用代码最小 CI 的默认执行顺序应为：
1. 先执行静态与结构校验层，尽早阻断格式、边界或最低依赖检查问题
2. 再执行测试校验层，验证单元测试与架构测试构成的默认测试集合
3. 最后执行可运行性与构建确认层，验证运行时装配与最小可运行前提

这种顺序用于提供平台无关的流水线映射基础：前一层失败时，不应继续执行后一层；后续若映射到具体 CI 平台，可直接将三层映射为串行 stage 或等价阻断作业，而无需改变本文定义的职责边界。

## 与 `docs/cicd_baseline.md` 的关系
- `docs/cicd_baseline.md` 负责平台中立的 CI/CD 主基线、默认门禁框架与受控部署边界
- 本文负责应用代码的默认覆盖范围、三层最小流水线模型与默认阻断门禁
- 后续若需要映射到具体平台流水线，应在不破坏两者分层关系的前提下进行实现

## 当前仓库事实边界
- 当前仓库的 `pyproject.toml` 已声明 `uv` 为依赖管理与命令执行基线
- 当前仓库已具备 `app/` 应用代码骨架与 `tests/unit/`、`tests/integration/`、`tests/architecture/` 测试分层基础
- 当前文档不把 GitHub Actions、GitLab CI、Jenkins 或其他平台资产写成现状
- 当前文档不把镜像扫描、签名、审批链、多环境晋升或企业平台能力写成现状

## 校验要点
- 是否明确限定应用代码最小 CI 的默认覆盖对象
- 是否采用三层最小流水线模型表达默认校验顺序与职责
- 是否将架构/边界检查列为默认阻断项
- 是否将默认测试集合限制在与现有测试分层一致的最小范围
- 是否把可运行性与构建确认层限制在最小可运行前提
- 是否避免将文档门禁、CD 或平台配置写入本文范围

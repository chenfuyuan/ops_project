# CI/CD 基线

## 文档目标
本文档定义项目的 CI/CD 基线，用于在平台中立前提下约束代码变更从校验、构建到受控部署的最小交付闭环。

## 适用范围
- 代码与文档变更的默认校验流程
- 可发布制品的生成与交付约束
- 受控部署边界
- 发布后最小验证与回滚入口

## 本文负责 / 不负责
### 本文负责
- 定义平台中立的 CI/CD 阶段模型
- 定义默认 CI 门禁
- 定义可发布制品的最小要求
- 定义受控部署的最小闭环
- 定义与发布、安全、部署文档的职责映射

### 本文不负责
- 不定义具体 CI/CD 平台语法或配置文件
- 不定义多环境晋升、复杂灰度、蓝绿或金丝雀策略
- 不定义云资源拓扑、值班 Runbook 或平台操作手册
- 不假定镜像仓库、签名、扫描、审批链等企业平台能力已经落地

## CI/CD 阶段模型
### 校验阶段
用于验证变更是否满足默认交付准入条件。

至少包括：
- 格式或静态检查
- 测试
- 必要的构建校验
- 文档一致性校验
- 架构/边界检查
- 依赖/安全最低门禁

### 构建阶段
用于在通过默认门禁后生成可发布制品，或确认本次变更对应的可发布交付对象。

要求：
- 生成结果可追踪到具体变更
- 可与发布记录、验证结果和回滚入口关联
- 不因平台差异改变其作为交付对象的语义

### 受控部署阶段
用于在满足部署触发与目标环境准入条件后，将可发布制品推进到目标环境。

至少包括：
- 部署触发条件
- 目标环境准入条件
- 发布后最小验证
- 回滚入口

## 默认 CI 门禁
### 统一要求
默认 CI 门禁是交付准入条件的一部分，而不是可选增强项。

### 最低门禁清单
- 格式或静态检查通过
- 相关测试通过
- 必要的构建校验通过
- 文档与当前变更保持一致
- 架构/边界检查通过
- 依赖与安全最低检查通过

### 架构/边界门禁要求
架构/边界检查必须作为默认阻断项处理，不能仅作为提示信息。

### 文档一致性门禁要求
文档一致性校验必须作为默认阻断项处理，不能仅作为发布说明补充或参考项。

### 依赖/安全最低门禁要求
依赖/安全最低门禁必须作为默认阻断项处理，不能仅作为后续增强项。

该要求应与现有架构规则和测试策略保持一致，例如：
- 依赖方向约束
- 业务边界不直接依赖实现细节
- `interfaces`、`capabilities`、`shared` 的越界依赖检查

## 可发布制品
### 定义
可发布制品是交付闭环中的核心对象。它必须：
- 可追踪
- 可验证
- 可与发布记录和回滚入口关联

### 表达要求
- 容器镜像仅作为优先兼容的制品形式
- 但镜像不是唯一合法交付物
- 当仓库尚未落地某类制品治理能力时，不得将其写成现成事实

## 受控部署边界
受控部署闭环只覆盖部署触发、目标环境准入、发布后最小验证与回滚入口，不扩展到多环境晋升或复杂发布编排。

### 部署触发
只有在默认 CI 门禁通过并满足目标环境准入条件后，才应进入部署。

### 环境准入
环境准入应与 `local / test / dev / prod` 的环境分层保持一致，并明确：
- 哪些变更可进入目标环境
- 发布前需要满足哪些最小条件
- 哪些验证结果应作为部署输入

### 最小发布后验证
部署完成后至少应验证：
- 目标入口基本健康
- 关键链路未出现明显异常
- 后台处理路径在需要时可正常工作
- 未出现明显新增错误日志、错误率或任务积压问题

### 回滚入口
受控部署必须存在明确回滚入口，使交付对象能够与回退动作关联。
复杂发布策略不属于本文范围。

## 文档职责映射
### 本文与 `docs/application_minimal_ci_baseline.md`
- 本文负责平台中立的 CI/CD 主基线、默认门禁框架与受控部署边界
- 应用代码最小 CI 实践文档负责收敛 `app/` 与直接相关测试的默认覆盖范围、三层最小流水线模型与默认阻断门禁
- 两者应保持分层协作，不应由主基线文档继续承接下游应用代码实践细节

### 本文与 `docs/release_change_and_rollback.md`
- 本文负责 CI/CD 交付闭环与阶段边界
- 发布与回滚文档负责发布控制、验证与回滚原则的专题基线

### 本文与 `docs/security_baseline.md`
- 本文负责把依赖/安全最低门禁纳入默认 CI 门禁
- 安全文档负责安全原则、配置安全、制品与变更安全要求的专题基线

### 本文与 `docs/deployment_environment_baseline.md`
- 本文负责定义受控部署所需的环境准入边界
- 部署环境文档负责环境分层、最小组件和部署基线

## 当前仓库事实边界
- 当前文档只定义平台中立的能力模型与治理约束
- 应用代码的默认覆盖范围、三层最小流水线模型与默认阻断门禁由 `docs/application_minimal_ci_baseline.md` 承接
- 不预设 GitHub Actions、GitLab CI、Jenkins 或其他平台实现已经存在
- 不把镜像扫描、签名、审批链、多环境晋升或云平台交付资产写成现状
- 当仓库后续落地具体平台资产时，应在不改变本文边界的前提下进行平台映射

## 校验要点
- 是否以平台中立方式表达 CI/CD 阶段模型
- 是否将架构/边界检查列为默认阻断门禁
- 是否将可发布制品定义为交付闭环中心对象
- 是否把 CD 范围限制在受控部署闭环内
- 是否避免将未落地的平台能力写成现成事实
- 是否保持与发布、安全、部署文档的职责分层

## 例外与变更机制
如需引入具体平台实现、多环境晋升策略或更复杂的发布模型，应在不破坏本文边界的前提下通过专门文档或后续 change 补充。

# 部署与环境基线

## 文档目标
本文档定义项目在企业级交付场景下的环境分层、最小部署形态和环境治理基线。

## 适用范围
- 本地开发环境
- 测试环境
- 开发联调环境
- 生产环境

## 本文负责 / 不负责
### 本文负责
- 定义 `local / test / dev / prod` 环境分层
- 定义各环境的最小组件组成
- 定义部署基线和环境差异控制原则
- 定义环境治理的最低要求

### 本文不负责
- 不展开密钥管理细则，详见 `docs/configuration_and_secrets_management.md`
- 不展开告警与监控细则，详见 `docs/observability_and_alerting.md`
- 不展开发布流程细则，详见 `docs/release_change_and_rollback.md`
- 不重新定义系统边界，详见 `docs/technical_architecture_baseline.md`

## 环境分层定义
### local
用于开发者本地开发、调试和基础验证。

要求：
- 允许使用 Docker Compose 启动依赖
- 可使用精简数据集或本地假数据
- 保持与正式环境一致的配置模型

其中，Docker Compose 表达的是推荐的本地编排方式，用于保持 API、Worker 与基础依赖的运行拓扑一致；这不等同于当前仓库已经提交了对应的 Compose 文件或容器镜像定义。

### test
用于自动化测试、集成测试和回归验证。

要求：
- 可重复初始化
- 测试数据可销毁
- 不与开发联调环境共用数据库和密钥

### dev
用于多人联调、接口联通和部署验证。

要求：
- 具备接近正式环境的组件拓扑
- 支持 API 与 Worker 联动验证
- 支持迁移、对象存储、缓存和后台任务联调

### prod
用于正式业务承载。

要求：
- 变更受控
- 配置与密钥独立管理
- 有明确的健康检查、告警和回滚能力

## 最小运行组件
所有正式环境至少应包含以下组件：

| 组件 | 作用 |
| --- | --- |
| API Process | 承载同步请求入口与任务投递 |
| Worker Process | 承载耗时、可重试和后台任务 |
| PostgreSQL | 承载核心业务数据与结构化元数据 |
| Redis | 承载缓存、轻量协调与队列支撑 |
| S3-compatible Object Storage | 承载文件和大对象产物 |

该最小拓扑与 `docs/technical_architecture_baseline.md` 保持一致。

## 部署基线
### 统一原则
- 默认以容器化部署作为正式环境的交付基线
- API 与 Worker 必须可独立部署和扩缩
- 数据库迁移应作为独立步骤执行
- 环境差异应主要体现在配置、容量和访问控制，而不是运行模型
- 当前仓库若尚未提供 `Dockerfile`、Compose 文件或平台部署清单，不应在评审或文档中将这些工程产物描述为已存在

### API 基线
- 必须提供健康检查入口
- 必须能注入独立环境配置
- 不应在 API 进程中承担长期后台任务

### Worker 基线
- 必须具备独立的启动与停止方式
- 必须具备最小可观测能力
- 不应承担协议入口职责

### 存储基线
- PostgreSQL、Redis、对象存储在正式环境中应为独立的环境资源
- 不同环境之间不得共用生产数据与生产密钥

## 环境差异控制原则
- 环境差异优先通过配置注入体现
- 不允许在业务代码中写死环境分支
- 本地、测试、开发、生产应尽量共享同一套装配模型
- 实现选择应集中在 `bootstrap` 完成，而不是渗透到业务逻辑
- 当项目按 `docs/cicd_baseline.md` 组织交付时，目标环境准入属于受控部署的一部分，并与本文定义的环境分层保持一致，而不是扩展为多环境晋升编排或平台实现细节

## 角色与责任
- 开发：保证代码和迁移适配统一部署模型
- 评审者：检查是否引入环境耦合或部署前置假设
- 运维或发布责任人：维护环境配置、访问控制和资源可用性

## 校验要点
- 是否统一使用 `local / test / dev / prod` 环境名称
- 是否明确 API、Worker、PostgreSQL、Redis、对象存储的环境职责
- 是否避免在业务代码中引入环境分支
- 是否将迁移、配置注入和部署步骤视为独立治理对象
- 是否避免把尚未提交到仓库的 Docker/Compose/部署清单描述为现成资产
- 是否将目标环境准入条件与受控部署边界分层表达，而不是混写为平台实现细节

## 例外与变更机制
如需引入新的环境层级、改变最小拓扑或绑定特定云平台，应在评审中明确说明原因、影响范围和替代方案，并同步更新本文档。

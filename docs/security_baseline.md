# 安全基线

## 文档目标
本文档定义项目的最低安全要求，覆盖应用安全、数据安全、配置安全、依赖安全与安全例外管理。

## 适用范围
- API 与 Worker 应用层
- 配置与密钥管理
- 数据库存储与对象存储访问
- 构建、依赖与发布变更过程

## 本文负责 / 不负责
### 本文负责
- 定义统一安全原则
- 定义应用层、数据层和依赖层的最低安全要求
- 定义日志脱敏与审计约束
- 定义安全例外管理要求

### 本文不负责
- 不替代具体的鉴权业务设计文档
- 不展开密钥注入流程，详见 `docs/configuration_and_secrets_management.md`
- 不展开发布步骤，详见 `docs/release_change_and_rollback.md`

## 统一安全原则
- 最小权限
- 默认拒绝
- 敏感信息不入库、不入日志、不入业务核心
- 边界清晰，技术细节不回流到业务核心
- 高风险变更必须可审查、可追踪、可回滚

## 应用层安全要求
- 所有外部输入都应在系统边界进行校验
- 鉴权和授权逻辑应明确位于协议边界或业务明确约定的位置
- 不应在业务核心中散落处理原始认证令牌或底层权限实现细节
- 远程调用鉴权、签名、超时和重试应留在基础设施适配层

## 数据与存储安全
- PostgreSQL 承载核心业务数据和可审计元数据
- Redis 不应作为唯一业务事实来源保存关键安全状态
- 对象存储中的文件访问应受控，避免公开暴露默认可读对象
- 涉及敏感字段时，应优先保存必要最小信息

## 配置与密钥安全
- 密钥和凭据不得硬编码
- 不同环境必须隔离密钥
- 调试信息不得输出敏感配置原文
- 详见 `docs/configuration_and_secrets_management.md`

## 依赖与供应链基线
- Python 依赖以 `pyproject.toml` 和 `uv.lock` 为单一事实来源
- 依赖升级应受评审控制
- 不引入来源不明或维护状态不清晰的核心依赖
- 遇到安全漏洞时，应优先评估影响范围、缓解措施和升级计划

## 日志与审计安全
- 日志必须对密钥、凭据、令牌和敏感连接信息做脱敏
- 关键变更应可追踪到责任人和时间点
- 高风险操作应保留足够的上下文用于事后审查

## 安全例外管理
当出现无法立即满足基线的情况时，应记录：
- 例外原因
- 影响范围
- 临时补偿措施
- 责任人
- 失效时间或清理时间

## 角色与责任
- 开发：保证新增代码遵守输入校验、敏感信息保护和依赖控制原则
- 评审者：检查越权访问、敏感信息暴露和边界破坏风险
- 发布责任人：确保高风险变更有审查和回滚准备

## 校验要点
- 是否在系统边界处理外部输入校验
- 是否避免在业务核心中处理底层安全实现细节
- 是否对日志和配置进行了脱敏要求
- 是否对依赖和高风险变更设置了最小控制要求

## 例外与变更机制
如因第三方限制、遗留系统约束或短期交付压力需要暂时突破安全基线，必须形成明确记录，并在例外结束后回收。

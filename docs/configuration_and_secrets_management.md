# 配置与密钥管理

## 文档目标
本文档定义项目的配置分类、配置来源、密钥保护和变更控制基线，确保不同环境中的配置可管理、可审计、可替换。

## 适用范围
- 应用运行配置
- 基础设施连接配置
- 业务参数配置
- 密钥、令牌、凭据等敏感配置

## 本文负责 / 不负责
### 本文负责
- 定义配置分类与来源优先级
- 定义密钥管理的最低要求
- 定义配置注入、轮换与审计原则
- 定义禁止项和反模式

### 本文不负责
- 不定义具体部署平台的 Secret 产品选型
- 不展开统一安全总则，详见 `docs/security_baseline.md`
- 不展开发布步骤，详见 `docs/release_change_and_rollback.md`

## 配置分类
### 通用运行配置
例如：运行模式、日志级别、服务名、端口、超时阈值。

### 基础设施连接配置
例如：数据库地址、Redis 地址、对象存储地址、消息队列连接参数。

### 业务参数配置
例如：任务批大小、限流阈值、业务开关、默认策略参数。

### 敏感配置
例如：数据库密码、访问令牌、签名密钥、第三方服务凭据。

## 配置来源与优先级
推荐采用以下优先级：
1. 运行时环境变量或等价的外部注入机制
2. 启动阶段装配的类型化配置对象
3. 稳定的默认值

要求：
- 配置读取应集中在 `bootstrap` 或 `shared/infra` 的配置加载层
- 业务核心不应散落读取原始环境变量
- 配置字段应具备明确命名和类型约束

## 密钥管理基线
- 禁止将密钥硬编码在代码中
- 禁止将密钥提交到 Git 仓库
- 禁止将密钥打印到日志中
- 禁止将密钥直接暴露给业务核心对象
- 不同环境必须使用隔离的密钥集合
- 密钥权限应遵循最小权限原则

## 注入与轮换原则
- 密钥应在部署或运行时注入，而不是在代码中内置
- 密钥轮换应不依赖业务逻辑改造
- 当凭据即将失效或权限范围变化时，应有明确变更窗口和回滚准备
- 应避免把长期凭据分发到个人开发环境中作为默认做法

## 审计与排查要求
- 配置项变更应可追踪
- 敏感配置访问应受控
- 排查问题时，可以输出配置项是否存在、是否通过校验，但不得输出密钥原文
- 需要调试时，应优先输出脱敏后的配置信息

## 角色与责任
- 开发：通过类型化配置对象消费配置，避免业务层直接读取环境变量
- 评审者：检查是否出现硬编码、日志泄漏和环境耦合
- 发布责任人：保证配置来源、密钥注入和轮换流程可执行

## 反模式清单
- 在 `service.py` 中直接读取环境变量
- 在仓库中提交明文凭据
- 在异常日志中输出完整连接串或密钥
- 将本地开发密钥复用到测试或生产环境
- 用多个真相来源同时管理同一配置

## 校验要点
- 是否对配置做了分类管理
- 是否明确了运行时注入优先于代码内置
- 是否明确了密钥不得入库、入日志、入业务核心
- 是否把配置读取集中在装配层或基础设施层

## 例外与变更机制
如确需引入特殊配置来源或短期凭据方案，应记录适用范围、过期时间、补偿措施和责任人，并在方案失效后清理。

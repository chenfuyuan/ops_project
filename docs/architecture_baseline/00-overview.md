## 1. 本文件的定位

本文件是拆分后的总入口，用于帮助 AI 或开发者快速判断：

- 这套架构基线解决什么问题；
- 顶层结构如何理解；
- 遇到具体问题时应该优先读取哪一份拆分文档。

本文件只保留总览级信息，不展开节点内部细节、反腐实现细节、治理细则和测试细则。

---

## 2. 架构目标

该架构基线用于定义一个**不包含任何业务逻辑**的新项目骨架，适合用于：

- 基于工作流编排的后端项目；
- 需要沉淀可复用能力组件的项目；
- 当前以单体落地、未来可演进为独立服务或微服务的项目。

它重点回答以下问题：

1. 边界如何定义；
2. 依赖如何约束；
3. 防腐层如何落地；
4. 如何支持未来演进；
5. 如何防止目录和模块逐步腐化。

本基线不讨论：

- 具体业务领域；
- 具体业务流程细节；
- 领域规则实现；
- 厂商绑定和具体三方服务选型。

## 4. 顶层目录总览

推荐目录如下：

```text
project_scaffold/
├─ app/
│  ├─ business/
│  ├─ capabilities/
│  ├─ interfaces/
│  ├─ shared/
│  └─ bootstrap/
```

顶层边界的核心语义如下：

- `business`：唯一允许承载业务语义的主区域；
- `capabilities`：承载无业务语义、跨业务复用的通用能力；
- `interfaces`：承载外部协议入口；
- `shared`：承载稳定、低业务语义、跨上下文可复用的基础模块；
- `bootstrap`：承载运行时装配与实现选择。

详细定义见 `01-top-level-modules.md`。

## 5. 核心架构目标

该架构基线要求同时满足以下目标：

- 边界清晰：业务语义、通用能力、外部协议、共享基础设施不混淆；
- 结构轻量：前期不强制重型分层；
- 依赖可控：通过显式端口和适配约束调用方向；
- 可替换：本地调用可平滑切换为 HTTP / RPC / MQ；
- 可复用：真正通用的能力沉淀到 `capabilities`；
- 可演进：单体形态可逐步演进为独立服务；
- 可测试：业务模块依赖抽象端口；
- 可治理：通过准入规则和自动化约束防止结构腐化。

## 6. 核心设计原则

### 6.1 业务优先，技术从属

目录结构优先表达业务边界和系统边界，而不是优先表达技术类型。

### 6.2 先平铺，后演进

节点内部前期采用轻量平铺结构，通过文件职责区分；只有复杂度显著上升时，才继续细分层次。

### 6.3 通用能力必须无业务语义

`capabilities` 中只允许放跨业务复用、输入输出已规范化、且不带业务属性的通用能力。

### 6.4 业务必须拥有自己的依赖边界

业务通过 `ports` 定义所需能力，再由 `infrastructure` 提供实现，避免业务直接依赖具体调用方式。

### 6.5 防腐层是边界实现，不是额外口号

防腐层通过业务侧 `ports + infrastructure` 落地：

- `ports` 定义业务语言中的接口；
- `infrastructure` 负责把业务语言翻译成外部世界或通用能力的调用。

### 6.6 共享模块必须克制

`shared` 只承载稳定、通用、低业务语义的内核与基础设施能力，不能成为杂物间。

### 6.7 面向部署形态变化设计

今天是本地类调用，未来可能是远程服务；业务侧不应感知这种变化。

### 6.8 准入比归档更重要

判断某段代码“能否进入某层”，比“暂时放在哪个目录”更重要。

### 6.9 边界必须可执行

边界需要通过命名约束、依赖检查、测试替身、装配规则和代码审查持续守护。

## 8. 使用建议

如果你是 AI，在回答问题时建议遵循以下方式：

- 判断“代码应该放哪”时，先看顶层模块语义，再看业务内部文件职责；
- 判断“能不能直接调用”时，先检查是否越过了 `ports` 和 `infrastructure`；
- 判断“是否该抽成通用能力”时，优先检查是否满足跨业务复用、无业务语义、输入输出可标准化；
- 判断“是否属于共享层”时，优先检查是否稳定、低语义、跨上下文长期复用。

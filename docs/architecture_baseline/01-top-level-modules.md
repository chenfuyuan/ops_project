# 顶层模块职责与边界

- 来源文档：`docs/new_project_architecture_baseline_clean.md`
- 关联文档：`00-overview.md`

---

## 1. 本文件回答的问题

本文件用于回答以下问题：

- 顶层为什么要划分为 `business / capabilities / interfaces / shared / bootstrap`；
- 每个顶层模块负责什么、不负责什么；
- 哪些依赖允许，哪些依赖禁止；
- 哪些内容可以进入 `capabilities` 或 `shared`。

本文件不展开业务节点内部文件职责，相关内容见 `02-business-internal-structure.md`。

---

## 2. 顶层结构

推荐目录如下：

```text
project_scaffold/
├─ app/
│  ├─ business/
│  ├─ capabilities/
│  ├─ interfaces/
│  ├─ shared/
│  └─ bootstrap/
```

顶层划分的核心目标是让系统边界先于技术实现暴露出来。

---

## 3. `business/`

`business` 是系统中唯一允许承载**业务语义**的主区域。

### 职责

- 定义业务工作流；
- 组织业务节点；
- 定义业务对象、业务规则、业务用例；
- 定义业务需要的能力端口 `ports`；
- 通过 `infrastructure` 实现对外部世界和通用能力的适配；
- 在业务域内部沉淀有限范围的共享业务能力。

### 不负责

- 直接向其他模块暴露三方 SDK；
- 通用能力的跨业务抽象沉淀；
- HTTP、消息订阅等协议层入口处理；
- 系统级启动装配。

### 允许依赖

- `shared`；
- 自身业务模块内部的 `ports` / `infrastructure`；
- 通过业务侧 `infrastructure` 间接对接 `capabilities`。

### 禁止事项

- 被 `shared` 反向依赖；
- 业务域之间通过实现细节互相耦合；
- `service.py`、`rules.py`、`entities.py` 直接依赖 `capabilities` 的具体实现；
- 业务代码直接依赖供应商 SDK、HTTP client、ORM 具体实现。

### 关键说明

“`business` 可以使用 `capabilities`”并不意味着业务实现可以直接调用 capability 的具体实现。推荐调用链应始终保持为：

```text
service.py -> ports.py -> infrastructure/* -> capabilities/*
```

这样可以保证业务语言稳定，不被能力层接口、供应商字段或调用协议直接污染。

---

## 4. `capabilities/`

`capabilities` 用于承载**跨业务复用的通用能力组件**。

### 职责

- 对外提供无业务语义的标准能力接口；
- 屏蔽第三方服务或底层实现差异；
- 统一输入输出模型；
- 为未来独立部署或微服务化保留空间。

### 典型能力形态

- 文本生成；
- 检索查询；
- 内容抓取；
- 文件解析；
- 通用审核；
- 统一通知通道。

### 允许依赖

- `shared`；
- 自身内部的 `infrastructure`。

### 禁止事项

- 依赖 `business`；
- 依赖 `interfaces`；
- 依赖 `bootstrap`；
- 以“通用组件”名义下沉业务规则；
- 对业务层暴露供应商特定字段。

### 准入标准

只有同时尽量满足以下条件的内容，才适合进入 `capabilities`：

- 存在明确的跨业务复用需求；
- 命名可以采用中性语言表达，而不是业务语义命名；
- 输入输出模型可以在能力层完成统一标准化；
- 外部供应商差异可以在能力内部吸收；
- 即使未来独立部署，也不会要求业务侧大规模改写语义接口。

如果某能力暂时不满足这些条件，应优先保留在具体业务域内，而不是为了“看起来通用”而提前平台化。

---

## 5. `interfaces/`

`interfaces` 用于承载所有外部协议入口。

### 职责

- 接收外部请求；
- 做协议层数据解析与响应组装；
- 进行鉴权、路由、协议映射、状态码转换等边界处理；
- 调用 `business` 暴露的应用入口。

### 不负责

- 业务规则判断；
- 业务对象持久化；
- 工作流编排细节；
- 外部依赖适配。

### 允许依赖

- `business`；
- `shared`。

### 禁止事项

- 直接依赖具体三方 SDK；
- 绕开业务入口直接调节点实现；
- 跨过业务边界直接进入 `business/**/infrastructure/*`。

---

## 6. `shared/`

`shared` 用于沉淀**稳定、低业务语义、跨上下文可共用**的基础模块。

### 职责

- 通用基础类型与内核对象；
- 通用异常基类、结果类型、标识符、时间、配置抽象；
- 事件基类与系统级事件基础设施；
- 与具体业务无关的基础设施组件。

### 子目录建议

#### `shared/kernel/`

放稳定的通用内核构件，例如：

- `Result`
- `DomainError`
- `Identifier`
- `Clock`
- `Pagination`
- 通用值对象基类

#### `shared/infra/`

放系统级基础设施抽象或实现，例如：

- 数据库连接管理；
- HTTP 客户端封装；
- 配置加载；
- 日志与链路追踪封装；
- 序列化基础设施；
- 重试、超时、熔断等通用调用策略封装。

#### `shared/events/`

放事件相关基础设施，例如：

- 事件基类；
- 事件发布器接口；
- 事件总线基础实现；
- Outbox 等公共机制。

### 严格禁止

- 任何带明显业务语义的对象进入 `shared`；
- 把“目前没地方放”的代码随手塞进 `shared`；
- 通过 `shared` 建立隐式跨业务耦合；
- 将业务 DTO、业务枚举、业务规则以“基础组件”名义沉入 `shared`。

### 准入标准

只有满足以下特征的内容才适合进入 `shared`：

- 与具体业务域无关；
- 可被多个上下文稳定复用；
- 生命周期预期较长，不因单一业务变化频繁波动；
- 抽象后不会反向要求业务语义向它妥协。

---

## 7. `bootstrap/`

`bootstrap` 负责系统装配与启动。

### 职责

- 依赖注入容器搭建；
- 运行时配置装配；
- 实现选择，本地 / HTTP / Mock / 第三方；
- 应用启动入口初始化。

### 不负责

- 业务逻辑；
- 业务规则；
- 协议处理细节；
- 通用能力实现细节本身。

### 边界规则

- `bootstrap` 可依赖全项目各模块用于装配；
- 其他模块不应反向依赖 `bootstrap`；
- 任何运行时实现切换应优先在 `bootstrap` 完成，而非侵入业务逻辑。

---

## 8. 顶层模块判断口诀

当你需要判断一段代码应该放哪里时，可以先按下面的顺序判断：

1. 它是否承载业务语义？如果是，优先看 `business`；
2. 它是否是无业务语义、可跨业务复用的能力？如果是，考虑 `capabilities`；
3. 它是否是协议入口或协议映射？如果是，放 `interfaces`；
4. 它是否是稳定、低语义、跨上下文复用的基础设施或内核？如果是，放 `shared`；
5. 它是否只负责装配与实现切换？如果是，放 `bootstrap`。

如果一个对象同时看起来像多个模块的候选，优先检查它是否携带业务语义。一旦携带业务语义，就不应进入 `capabilities` 或 `shared`。

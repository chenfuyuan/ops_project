# 新项目脚手架：纯骨架版模块化单体

## 一、目标

这是一份 **不包含任何具体组件、不包含任何业务知识** 的项目脚手架设计文档。

适用场景：

- 作为新项目起点
- 先做单体，后续可能拆分服务
- 希望从第一天就保留清晰模块边界
- 不想过早引入具体业务模块和平台组件

这份骨架的重点不是“现在有哪些模块”，而是：

- 未来任何模块都能按统一方式接入
- 模块之间有稳定边界
- 后续独立拆分服务时迁移成本低
- 当前仍保持单体开发效率

---

## 二、设计原则

### 1. 按模块纵向组织，而不是全局横向分层

不推荐一开始就使用这种全局目录：

```text
api/
domain/
application/
infrastructure/
```

因为这种结构更像经典分层单体，未来拆模块时，代码往往分散在多个顶层目录中，不容易整体迁移。

更推荐：

```text
modules/
  <module_a>/
  <module_b>/
```

每个模块内部再各自分层。

---

### 2. 模块之间只能通过公开接口通信

模块之间只能依赖对方暴露的：

- `api/contracts.py`
- `api/facade.py`

不能直接访问别人的：

- `application/`
- `domain/`
- `infrastructure/`

也不能直接访问别人的数据库表、ORM 模型、内部工具类。

---

### 3. 共享层必须极小

共享目录只能放真正通用、稳定、无业务语义的东西，例如：

- 基础异常
- ID 类型
- 通用配置
- 数据库连接
- 日志
- 事件总线基础接口

不能把共享层做成“大杂烩 common 包”。

---

### 4. 先设计边界，再逐步填充模块

脚手架阶段只解决：

- 项目怎么组织
- 模块怎么接入
- 依赖怎么管理
- 模块如何暴露公开能力

不预设任何具体业务组件。

---

### 5. 单体运行，边界按未来服务拆分标准设计

当前阶段：

- 所有模块运行在同一进程、同一仓库、同一部署单元中

未来阶段：

- 某个模块可以被替换成远程服务客户端，而不必重写调用方业务逻辑

---

## 四、各目录职责

## 4.1 `app/modules/`

这是整个项目最核心的目录。

它只负责承载“模块”。

在脚手架阶段，这里可以先不放任何具体模块，只放一个说明文件，约定未来模块的结构和接入方式。

未来任何模块都按统一模板接入。

### 模块标准结构

```text
app/modules/<module_name>/
├─ api/
│  ├─ contracts.py
│  └─ facade.py
├─ application/
│  ├─ commands.py
│  ├─ queries.py
│  └─ services.py
├─ domain/
│  ├─ entities.py
│  ├─ value_objects.py
│  ├─ rules.py
│  └─ exceptions.py
├─ infrastructure/
│  ├─ repositories.py
│  ├─ orm_models.py
│  └─ adapters.py
└─ module.py
```

### 模块各子目录职责

#### `api/`
模块的公开面。

只允许其他模块依赖这里。

- `contracts.py`：公开请求、响应、DTO、协议定义
- `facade.py`：对外暴露统一入口

#### `application/`
模块内部用例编排层。

负责：

- 组织业务用例
- 调用仓储、领域对象、外部适配器
- 处理内部流程

#### `domain/`
模块内部领域模型层。

负责：

- 实体
- 值对象
- 规则
- 领域异常

#### `infrastructure/`
模块内部基础设施实现层。

负责：

- 仓储实现
- ORM 模型
- 外部资源适配
- 第三方客户端封装

#### `module.py`
模块注册入口。

用于声明：

- 模块名称
- 模块对外 facade
- 生命周期钩子
- 初始化逻辑
- 需要注册的事件处理器

---

## 4.2 `app/interfaces/`

这是系统的输入输出适配层。

用于承载外部交互方式，不包含业务本体。

### `http/`
HTTP 接口层。

职责：

- 路由注册
- 请求响应模型
- 中间件
- 调用模块 facade

### `cli/`
命令行入口。

职责：

- 本地开发调试
- 管理任务
- 手工执行脚本命令

### `consumer/`
消息消费入口。

职责：

- 消费队列消息
- 作为异步任务入口
- 调用对应模块的公开能力

---

## 4.3 `app/shared/`

这里只放真正稳定的横向公共能力。

### `shared/kernel/`
最基础的共享内核。

建议只放：

- 基础类型
- 通用异常
- ID 封装
- Result 模型

### `shared/infra/`
基础设施公共封装。

建议只放：

- 配置读取
- 数据库连接
- 日志
- 缓存客户端
- 时间工具

### `shared/events/`
事件机制。

建议只放：

- 事件基类
- 事件总线接口
- 事件处理器注册逻辑

注意：

不要在这里放具体模块的事件定义。

---

## 4.4 `app/bootstrap/`

这是系统装配层。

负责把所有模块和系统基础设施组装起来。

### 典型职责

- 初始化配置
- 初始化数据库连接
- 注册模块
- 组装依赖注入容器
- 启动应用
- 绑定事件处理器

### 文件职责建议

- `container.py`：依赖注入容器
- `module_registry.py`：模块注册表
- `wiring.py`：模块间装配逻辑
- `startup.py`：启动初始化逻辑

---

## 4.5 `app/main.py`

应用启动入口。

职责：

- 创建应用对象
- 调用 bootstrap 逻辑
- 注册 HTTP 路由或其他接口层

---

## 4.6 `tests/`

测试建议分层：

### `tests/unit/`
单元测试。

聚焦：

- 纯函数
- 领域规则
- application 层小范围逻辑

### `tests/integration/`
集成测试。

聚焦：

- 模块内部完整调用链
- 数据库、缓存、外部适配器集成

### `tests/e2e/`
端到端测试。

聚焦：

- 从接口层打到系统内部
- 验证完整链路

---

## 4.7 `migrations/`

数据库迁移目录。

即便当前还是单库，也建议保留这个目录。

后续每个模块的数据表仍然应按模块归属设计，而不是随意混合。

---

## 4.8 `scripts/`

开发与运维辅助脚本目录。

例如：

- 初始化环境
- 本地启动
- 清理测试数据
- 生成示例配置

---

## 五、模块接入模板

当未来新增一个模块时，建议按下面的最小模板创建。

```text
app/modules/<module_name>/
├─ api/
│  ├─ contracts.py
│  └─ facade.py
├─ application/
│  └─ services.py
├─ domain/
│  └─ exceptions.py
├─ infrastructure/
│  └─ repositories.py
└─ module.py
```

这是最小可用模块。

后续再按复杂度逐步增加：

- `entities.py`
- `value_objects.py`
- `orm_models.py`
- `adapters.py`
- `commands.py`
- `queries.py`

## 六、模块间依赖规则

这是整个脚手架最关键的部分。

### 规则 1：模块间只能依赖公开 API

允许：

```python
from app.modules.some_module.api.facade import SomeModuleFacade
```

不允许：

```python
from app.modules.some_module.application.services import SomeInternalService
from app.modules.some_module.infrastructure.repositories import SomeRepository
```

---

### 规则 2：禁止跨模块直连数据库表

允许：

- 通过对方模块 facade 获取需要的数据或执行能力

不允许：

- 直接查询其他模块的 ORM 模型
- 直接操作其他模块内部表

---

### 规则 3：禁止跨模块共享内部 DTO

模块对外的数据结构只能来自：

- `api/contracts.py`

不要把内部领域对象直接暴露给外部模块。

---

### 规则 4：模块只暴露一个稳定入口

推荐每个模块只暴露一个 facade。

例如：

```python
class ModuleFacade(Protocol):
    ...
```

这样未来：

- 可以替换实现
- 可以切换为远程调用客户端
- 可以保持调用方不变

---

### 规则 5：共享层不得承载业务逻辑

不要把“暂时不知道放哪”的代码都塞进 `shared/`。

这是模块化单体最容易失控的点。

---

## 七、未来可拆分设计

虽然当前是单体，但骨架应支持未来将某个模块独立拆分出去。

### 当前阶段

模块之间是进程内调用：

```text
Module A -> Module B Facade（本地实现）
```

### 未来阶段

某个模块可以替换为远程客户端：

```text
Module A -> Module B Facade（远程 HTTP / RPC 客户端实现）
```

调用方代码不变，只替换装配逻辑。

### 为了支持这一点，建议：

- facade 接口稳定
- contracts 独立清晰
- 模块内部实现不要泄漏到外部
- 装配逻辑集中在 `bootstrap/`

---

## 八、推荐启动顺序

如果你现在要基于这个脚手架起一个新项目，推荐按下面顺序搭建。

### 第 1 步：搭好纯骨架

先创建：

- `modules/`
- `interfaces/`
- `shared/`
- `bootstrap/`
- `tests/`

此时不放任何业务模块。

### 第 2 步：完善共享基础设施

补齐：

- 配置管理
- 日志
- 数据库连接
- 应用启动入口
- 基础健康检查接口

### 第 3 步：定义模块模板

写好 `modules/README.md`，约定未来模块统一结构。

### 第 4 步：新增第一个真实模块

新增时严格按模块模板接入。

### 第 5 步：所有功能都通过模块扩展，而不是向全局目录追加代码

这能避免项目重新退化成普通分层单体。

## 九、推荐技术栈

这部分同样保持“纯骨架”思路，不绑定任何业务组件。

### 必选基础

- **Python 3.12+**：主语言
- **FastAPI**：HTTP 接口层
- **Pydantic**：数据模型与配置模型
- **SQLAlchemy 2.0**：ORM / 数据访问
- **Alembic**：数据库迁移
- **PostgreSQL**：默认关系型数据库
- **pytest**：测试框架
- **ruff**：lint 与格式检查
- **Docker**：运行环境封装

### 可选基础

- **Redis**：缓存 / 队列 / 幂等控制
- **httpx**：HTTP 客户端
- **structlog** 或标准 `logging`：结构化日志
- **mypy**：类型检查
- **pre-commit**：提交前检查

### 当前阶段不强制引入

- 消息队列
- 工作流引擎
- 向量数据库
- 服务注册发现
- 微服务网关
- 分布式追踪全家桶

这些都可以等项目真正需要时再接入。

## 十一、脚手架的核心价值

这套纯骨架版脚手架，不解决任何具体业务问题。

它解决的是更基础的问题：

- 新项目从哪里开始搭
- 新模块以后往哪里放
- 模块之间怎么通信
- 什么代码可以共享
- 什么代码不能乱共享
- 未来如何从单体平滑演进到独立服务

---

## 十二、一句话总结

这是一套 **不带业务、不带组件、只保留边界与规则** 的模块化单体脚手架。

它的目标不是现在就做复杂系统，而是：

**让任何新项目从第一天开始，就具备长期可演进、可拆分、可复用的工程基础。**

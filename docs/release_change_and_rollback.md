# 发布、变更与回滚

## 文档目标
本文档定义项目的发布、变更控制和回滚基线，确保应用、配置和数据库变更具备最小可控性。

## 适用范围
- 应用版本发布
- 配置变更
- 数据库迁移变更
- 与运行环境直接相关的交付步骤

## 本文负责 / 不负责
### 本文负责
- 定义变更类型
- 定义发布前检查项
- 定义发布、验证和回滚的最低流程
- 定义数据库迁移的控制原则

### 本文不负责
- 不定义具体 CI/CD 平台实现
- 不替代运维值班 Runbook
- 不替代各专题文档中的环境、密钥和告警细则

## 变更类型
### 应用变更
代码、依赖、接口、业务实现和后台任务逻辑变更。

### 配置变更
环境变量、连接参数、阈值和开关类配置变更。

### 数据库迁移变更
Schema 变更、索引变更、约束变更和数据迁移。

## 发布前检查
正式发布前至少检查以下事项：
- 相关测试已完成
- 配置项和密钥已准备并校验
- 数据库迁移策略已确认
- 发布后观察项已明确
- 回滚方案已明确
- 若采用容器化交付，应明确本次发布涉及的镜像版本、运行配置和需要同步变更的进程角色
- 不得把仓库中尚未落地的 Docker / Compose / 部署清单写成默认发布流程

## 发布流程基线
1. 准备发布内容与变更说明
2. 执行必要的数据库迁移
3. 部署应用变更
4. 验证 API 与 Worker 基本健康
5. 观察关键日志、指标和队列状态
6. 确认发布完成或进入回滚

在容器化交付场景下，镜像构建结果、镜像版本、运行配置和进程角色拆分也属于发布对象的一部分。
如果当前仓库仅落地了最小 Docker 基线，则发布与回滚流程应围绕 `api`、`worker` 与其已实现依赖形成最小闭环，不应把尚未实现的迁移服务、PostgreSQL、对象存储或平台清单描述为现成流程。

## 数据库迁移控制原则
- 所有 Schema 变更应通过 Alembic 管理
- 数据库迁移应作为独立治理步骤，不与应用启动强耦合
- 高风险迁移应提前评估兼容性、执行窗口和回滚方案
- 当无法安全回滚数据结构时，应在发布前明确这一限制

## 发布后验证
至少验证以下内容：
- API 健康检查通过
- 关键请求未出现异常错误率上升
- Worker 能正常消费任务
- 队列未出现异常积压
- 未出现新增的大量错误日志

## 回滚原则
### 应用回滚
- 当代码发布导致核心功能异常时，应优先支持回滚到上一个稳定版本
- 在容器化交付场景下，应用回滚通常体现为回退到上一稳定镜像版本或上一稳定容器配置

### 配置回滚
- 配置变更应具备恢复到前一版本的能力
- 高风险配置变更应独立记录

### 数据库回滚
- 数据库回滚应谨慎处理
- 对不可逆迁移，必须在执行前明确风险与补偿策略

## 变更记录与责任
- 每次正式发布应记录变更内容、时间和责任人
- 高风险变更应在评审阶段明确审批范围
- 发布期间应有明确的发布责任人

## 校验要点
- 是否区分了应用、配置和数据库迁移三类变更
- 是否定义了发布前检查、发布后验证和回滚原则
- 是否将数据库迁移视为独立控制对象
- 是否要求发布责任人与变更记录
- 在容器化发布场景下，是否区分了镜像、运行配置与进程角色变更
- 是否避免把仓库中尚未存在的 Docker / Compose / 部署资产写成默认发布流程

## 例外与变更机制
如需引入灰度、蓝绿、金丝雀或其他更复杂的发布模型，应在不破坏本文最低基线的前提下补充专门文档。

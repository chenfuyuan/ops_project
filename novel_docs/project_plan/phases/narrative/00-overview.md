# Phase 3 - Narrative

## Phase 目标

在一致性可靠之后，把重点转向“写得更好看”，围绕张力、类型化表达、知识组织和上下文层次提升叙事质量。

## Phase 边界

- 只讨论叙事质量相关需求，不回头承载一致性基线或风格控制的完整方案。
- 需求卡片保持在用户价值和成果标准层，不提前写出实现管线和技术选型。
- 与前端或 API 有关的需求，只表达能力入口，不展开接口设计。

## 进入标准

- Phase 2 的质量治理方向已经清晰，可以把主要风险从“写错”转向“写得不够好”。
- 团队需要更强的节奏控制、类型化写作和知识组织能力。

## 退出标准

- 本阶段每个核心能力已经形成可独立推进的 requirement card。
- 张力、类型化、知识图谱和上下文管理的职责分界足够清晰。
- 相关工作台需求已经能自然依附于这些核心能力，而不是单独漂浮。

## Phase 关系

- 前置：`consistency`
- 为后续解锁：`automation`

## 阅读路径

- 先看 [requirements.md](requirements.md)。
- 再针对最有价值的叙事能力进入对应卡片。

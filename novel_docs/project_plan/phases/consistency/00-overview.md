# Phase 2 - Consistency

## Phase 目标

在 MVP 可用之后，把“一章能写出来”提升为“长篇写作不容易自相矛盾”，建立一致性治理基线。

## Phase 边界

- 只关注审计、修订、事实保护、伏笔治理和质量追踪。
- 不把本阶段扩展成更广泛的叙事优化、文风塑造或自动驾驶设计。
- 卡片仍然只表达需求边界，不沉淀详细算法、数据结构或接口方案。

## 进入标准

- Phase 1 的最小闭环已经明确，团队开始遇到持续写长篇时的质量风险。
- 团队接受“先解决一致性，再谈更高级写作能力”的推进顺序。

## 退出标准

- 本阶段的需求已经拆成可独立进入 change 的卡片。
- 审计、修订、事实治理和伏笔治理的职责边界互不混淆。
- Phase 3 和 Phase 4 的能力前提已经在本阶段被清晰表达。

## Phase 关系

- 前置：`mvp`
- 为后续解锁：`narrative`, `style`, `automation`

## 阅读路径

- 先看 [requirements.md](requirements.md) 把握优先级和依赖。
- 再选择单张卡片决定是否继续进入 change。

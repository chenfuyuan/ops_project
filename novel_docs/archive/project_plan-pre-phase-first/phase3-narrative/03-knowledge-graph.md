# 知识图谱服务 (KnowledgeGraphService)

## 概述

自动从章节内容中提取主-谓-宾三元组，构建小说的知识图谱。支持信息边界查询（"角色 A 在第 N 章时知道什么"）和冲突检测。

## 三元组提取

### 提取时机

Reflector 在 Commit 阶段的 Delta 输出中增加三元组提取。

### 三元组字段

| 字段 | 类型 | 说明 |
|------|------|------|
| subject | str | 主语（角色名/地点名/物品名） |
| predicate | str | 谓语（持有/位于/攻击/知道/...） |
| object | str | 宾语 |
| tags | list[str] | 分类标签 |
| confidence | float | 置信度 0-1 |

### 标签分类

| 标签 | 含义 | 示例 |
|------|------|------|
| character | 涉及角色 | (陈墨, 攻击, 赵无极) |
| location | 涉及地点 | (黑风寨, 位于, 青云山北麓) |
| item | 涉及物品 | (陈墨, 持有, 火焰之剑) |
| relationship | 角色关系 | (陈墨, 师从, 林雪) |
| event | 事件 | (宗门大比, 发生于, 第50章) |
| ability | 能力 | (陈墨, 习得, 烈焰斩) |
| information | 信息传递 | (陈墨, 得知, 玉佩的秘密) |

### Reflector 提示词扩展

在 Reflector 提示词中追加知识图谱三元组提取要求：
- 提取所有角色行为、状态变化、关系变化、物品转移、位置移动
- 每个三元组标注分类标签
- 只提取明确发生的事实，不提取推测或计划
- 置信度：明确描述=1.0，暗示=0.7-0.9，模糊=0.5-0.7
- 输出为结构化 JSON 数组

## 信息边界查询

### 核心问题

"角色 A 在第 N 章时知道什么？"

这个查询对一致性审计至关重要——如果角色 A 在对话中提到了一个他不可能知道的信息，就是一个 critical 级别的矛盾。

### 查询逻辑

KnowledgeGraphService 的 query_character_knowledge 方法执行以下步骤：

1. 找到角色出场的所有章节（<= up_to_chapter）
2. 收集这些章节中角色在场时产生的三元组
3. 加上通过 "information" 标签传递给该角色的信息（predicate="得知"）
4. 去重后返回

check_knowledge_violation 方法用于检查角色是否有权知道某个信息，通过 LLM 判断 claimed_knowledge 是否能从已知三元组中推导出。

### 与 Auditor 的集成

Auditor 的"角色记忆"和"信息边界"维度直接调用知识图谱服务。对于章节中每个在场角色，查询其截至上一章的知识范围，检查章节中角色引用的信息是否在其知识范围内。违规项标记为 critical 级别的审计问题。

## 冲突检测

### 写入时检查

新三元组写入前自动检查与已有三元组的冲突。查找涉及相同主语和宾语的已有三元组，判断是否存在矛盾关系。

矛盾关系对包括：（持有, 失去）、（存活, 死亡）、（位于, 离开）、（信任, 背叛）、（同盟, 敌对）等。

### 冲突处理

- 时序合理的"冲突"不是真冲突（如先"持有"后"失去"）→ 正常写入
- 同一章节内的矛盾 → 标记为审计问题
- 跨章节的矛盾且无合理时序 → 标记为 critical 审计问题

## 前端

### 知识图谱可视化

使用力导向图（D3.js 或 ECharts graph）：
- 节点：实体（角色=圆形、地点=方形、物品=菱形）
- 边：关系（标注谓语）
- 节点大小：按关联三元组数量
- 颜色：按标签分类

### 交互

- 按角色过滤：只显示与某角色相关的子图
- 按章节范围过滤：只显示指定章节范围内的三元组
- 按标签过滤：只显示特定类型的关系
- 点击节点：显示该实体的所有三元组和来源章节
- 点击边：显示该关系的详情和来源

### 信息边界查询面板

- 选择角色 + 章节号
- 显示该角色在该时间点知道的所有信息
- 高亮标注信息来源（哪一章、通过什么方式得知）
